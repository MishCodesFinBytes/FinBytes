Instance Methods, Static Methods, and Top-Level Functions — Explained Properly

One of the most common sources of frustration when using pytest’s monkeypatch is this:

“Why is my patch not taking effect?”

The answer is almost always the same:

You are patching the wrong target.

This post will give you a precise mental model for identifying the correct monkeypatch target, covering:

  • Top-level functions

  • Instance methods

  • Static methods

  • Multiple classes with identical method names

  • Functions declared above classes

  • Direct imports vs module imports

By the end, you will know exactly how to determine the correct dotted path every time.

The Core Rule

You must patch the name where Python looks it up at runtime,
not where it was originally defined.

This distinction is critical.

Example Project Structure

project/
│
├── services.py
└── processor.py

services.py

def helper():
    return "real helper"


def top_level_function():
    return "real top level"


class A:
    def method(self):
        return "real A.method"

    @staticmethod
    def static_method():
        return "real A.static"

    def call_helper(self):
        return helper()


class B:
    def method(self):
        return "real B.method"

processor.py

from services import top_level_function, A
import services


def run_all():
    results = []

    results.append(top_level_function())
    results.append(A().method())
    results.append(A.static_method())
    results.append(services.B().method())
    results.append(A().call_helper())

    return results

Now we will determine the correct monkeypatch targets for each call.

1️⃣ Patching a Top-Level Function (Direct Import)

Inside processor.py:

from services import top_level_function

That means processor now has its own reference.

Correct patch:

monkeypatch.setattr("processor.top_level_function", mock_fn)

Incorrect:

monkeypatch.setattr("services.top_level_function", mock_fn)  # ❌

Because processor no longer looks it up via services.

2️⃣ Patching an Instance Method (Class Imported Directly)

Inside processor.py:

from services import A

Usage:

A().method()

Correct patch:

monkeypatch.setattr("processor.A.method", lambda self: "mocked A.method")

Note the self parameter — instance methods must accept it.

3️⃣ Patching a Static Method

Static methods behave differently in invocation,
but not in patching.

Usage:

A.static_method()

Correct patch:

monkeypatch.setattr(
    "processor.A.static_method",
    lambda: "mocked static"
)

Static methods do not require self.

4️⃣ Patching a Class Accessed Through Module Import

Inside processor.py:

import services

Usage:

services.B().method()

Now the lookup happens through services.

Correct patch:

monkeypatch.setattr(
    "services.B.method",
    lambda self: "mocked B.method"
)

Not "processor.B.method" — because B was never imported directly.

5️⃣ Patching a Function Declared Above the Class

Inside services.py:

def helper():
    return "real helper"


class A:
    def call_helper(self):
        return helper()

Important detail:

helper() is resolved inside the services namespace.

Even though A was imported into processor,
helper still belongs to services.

Correct patch:

monkeypatch.setattr("services.helper", lambda: "mocked helper")

Complete Test Example

def test_run_all(monkeypatch):
    import processor

    monkeypatch.setattr(
        "processor.top_level_function",
        lambda: "mocked top"
    )

    monkeypatch.setattr(
        "processor.A.method",
        lambda self: "mocked A.method"
    )

    monkeypatch.setattr(
        "processor.A.static_method",
        lambda: "mocked static"
    )

    monkeypatch.setattr(
        "services.B.method",
        lambda self: "mocked B.method"
    )

    monkeypatch.setattr(
        "services.helper",
        lambda: "mocked helper"
    )

    result = processor.run_all()

    assert result == [
        "mocked top",
        "mocked A.method",
        "mocked static",
        "mocked B.method",
        "mocked helper",
    ]

When Two Classes Have the Same Method Name

Consider:

class A:
    def execute(self):
        ...

class B:
    def execute(self):
        ...

You must patch each fully qualified path:

monkeypatch.setattr("processor.A.execute", ...)
monkeypatch.setattr("services.B.execute", ...)

Method names don’t matter.
The full dotted path does.

How to Derive the Correct Target (Step-by-Step)

When in doubt:

Step 1 — Open the file under test

Look at how the object is imported.

Step 2 — Look at how it is referenced

Is it func()?
Is it module.func()?
Is it Class.method()?

Step 3 — Patch exactly that lookup path

Debugging Trick

You can inspect what Python actually sees:

import processor

print(processor.A)
print(processor.top_level_function)
print(processor.A.__module__)

This reveals the real binding.

Common Mistakes

❌ Patching where defined instead of where used

This is the #1 error.

❌ Forgetting self in instance methods

lambda: "mocked"   # ❌
lambda self: "mocked"  # ✅

❌ Patching after object instantiation

If a class instance was created at import time:

obj = A()

Then patching A.method later may not affect that object.

The Heuristic to Memorize

Ask yourself:

Where is Python resolving this name at runtime?

Patch that.

Always.

Final Takeaway

Monkeypatching is not about where code lives.
It is about where code is looked up.

Once you internalize this resolution model, monkeypatch becomes predictable, deterministic, and safe — even across complex import graphs.
