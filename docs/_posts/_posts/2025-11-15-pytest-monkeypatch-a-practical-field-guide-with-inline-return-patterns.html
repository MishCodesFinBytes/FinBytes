Most engineers don’t struggle with what monkeypatch does.

They struggle with why it sometimes works perfectly — and sometimes appears to do nothing at all.

The difference is not pytest.

It is Python’s import system.

This post walks from first principles to advanced patterns and gives you a reliable mental model you can reuse in any codebase.

The Real Rule Behind Monkeypatch

Always patch where the dependency is looked up, not where it is defined.

If module1 imports a symbol from module2, your code never consults module2 again.
It uses its local reference.

So you patch:

module1.Class2.method

NOT:

module2.Class2.method

When tests mysteriously ignore your patch — this is almost always the cause.

Example Project (Runnable)

module2.py

class Class2:
    # library-style methods (no state)
    def multiply(x, y):
        return x * y

    def add(x, y):
        return x + y

    # instance-style method
    def instance_double(self, value):
        return value * 2

module1.py

from module2 import Class2

class Class1:
    def __init__(self):
        self.helper = Class2()

    def compute_library(self, a, b):
        return Class2.multiply(a, b) + Class2.add(a, b)

    def compute_instance(self, value):
        return self.helper.instance_double(value) + 5

Basic Monkeypatch Usage

Patching “library style” methods (no self)

def test_compute_library(monkeypatch):
    monkeypatch.setattr("module1.Class2.multiply", lambda x, y: 100)
    monkeypatch.setattr("module1.Class2.add", lambda x, y: 50)

    obj = Class1()
    assert obj.compute_library(3, 5) == 150

Patching instance methods (requires self)

def test_compute_instance(monkeypatch):
    monkeypatch.setattr(
        "module1.Class2.instance_double",
        lambda self, value: 999
    )

    obj = Class1()
    assert obj.compute_instance(10) == 1004

If you forget self you’ll get:

TypeError: <lambda>() takes 1 positional argument but 2 were given

Patching Only One Object

Sometimes you don’t want a global patch.

def test_single_instance(monkeypatch):
    obj = Class1()

    monkeypatch.setattr(
        obj.helper,
        "instance_double",
        lambda value: 500
    )

    assert obj.compute_instance(10) == 505

Because the method is already bound to the instance, self is no longer needed.

Inline Return Patterns (Quick Reference)

Cleaner Exception Pattern

Prefer a real function for readability:

def fail(*args, **kwargs):
    raise ValueError("forced failure")

monkeypatch.setattr("module1.Class2.multiply", fail)

Stateful Patch (Testing Retries)

def test_retry(monkeypatch):
    calls = {"n": 0}

    def fake(x, y):
        calls["n"] += 1
        if calls["n"] == 1:
            raise ValueError("fail once")
        return 100

    monkeypatch.setattr("module1.Class2.multiply", fake)

Now your test can verify retry logic.

Why Tests Sometimes Ignore Your Patch

Python import flow:

  1. module1 imports Class2

  2. Class1 stores reference locally

  3. Your code calls local reference

  4. Patching module2 changes nothing

You changed the source, not the pointer.

Monkeypatch modifies the pointer.

Lambda vs Function — When to Use Which

If it doesn’t fit on one line, don’t force a lambda.

Auto-Cleanup (Important)

monkeypatch automatically restores the original attribute after each test.

No teardown needed.
No leaking mocks.
No decorators required.

Mental Model to Remember

Monkeypatch is not mocking.

It is temporary attribute reassignment in a namespace.

Once you internalize that, 90% of confusion disappears.

Final Takeaway

Reliable monkeypatch usage comes down to three habits:

  1. Patch where the symbol is used

  2. Include self only when patching class-level instance methods

  3. Prefer inline lambdas for deterministic overrides

Everything else is composition.

After this, monkeypatch stops feeling magical —
and starts feeling predictable.
