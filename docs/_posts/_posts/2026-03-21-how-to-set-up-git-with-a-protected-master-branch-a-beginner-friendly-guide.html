A step-by-step walkthrough for connecting a local folder to GitHub, creating a working branch, and locking master so you never accidentally overwrite your clean code.

The Setup We're Building

If you've ever accidentally committed broken code to your main branch, this guide is for you. We're going to set up a simple but powerful workflow:

  • working branch → where you write and experiment freely

  • master branch → clean, stable, connected to GitHub — locked from direct edits

Prerequisites

  • Git installed on your machine

  • A GitHub account with an existing repo

  • Git Bash (Windows) or Terminal (Mac/Linux)

Step 1 — Clone Your GitHub Repo Locally

Navigate in your terminal to where you want the project to live, then run:

git clone https://github.com/your-username/your-repo.git

This downloads all your files and automatically sets up the remote connection. To confirm it worked:

git remote -v

You should see your GitHub URL listed against origin.

Step 2 — Create a Working Branch

Right now you're on master. Create a separate branch for all your day-to-day work:

git checkout -b working

You're now on the working branch. This is where all your editing happens from now on.

Step 3 — Make Your First Commit

Add your files and commit:

git add .
git commit -m "Initial commit"

Then push working up to GitHub:

git push origin working

Step 4 — Lock Master Locally Using a Git Hook

This is the key step. A git hook is a small script that runs automatically before every commit. We'll use it to block any commits directly to master.

Open Git Bash inside your repo folder (on Windows: right-click the folder → "Git Bash Here") and run:

cat > .git/hooks/pre-commit << 'EOF'
#!/bin/bash
branch=$(git symbolic-ref HEAD 2>/dev/null | sed 's|refs/heads/||')
if [ "$branch" = "master" ]; then
  echo "❌ Direct commits to master are not allowed. Use working branch."
  exit 1
fi
EOF

chmod +x .git/hooks/pre-commit

Test it works:

git checkout master
git commit --allow-empty -m "test"

You should see: ❌ Direct commits to master are not allowed. Use working branch.

Switch back to working:

git checkout working

Step 5 — Also Protect Master on GitHub

For an extra layer of protection, lock master on the remote too:

  1. Go to your repo on GitHub

  2. Click Settings → Branches → Add branch protection rule

  3. Set branch name pattern to master

  4. Tick "Require a pull request before merging"

  5. Save

Your Day-to-Day Workflow

From here on, this is your loop:

1. Write and edit code on the working branch
2. git add .
3. git commit -m "your message"
4. When ready to release → switch to master
5. git checkout master
6. git merge working
7. git push origin master
8. git checkout working  ← always come back here

Master stays clean. GitHub stays in sync. You never lose stable code.

Quick Command Reference

Common Gotchas

  • "pathspec did not match" — your branch probably doesn't exist yet because there are no commits. Run git add . && git commit -m "Initial commit" first.

  • Embedded repo warning — if you get a warning about an embedded git repository, check if there's a .git folder inside a subfolder and delete it.

  • main vs master — GitHub sometimes defaults to main. Check with git branch and update the hook script to match whichever name you're using.

Happy coding — and never fear the master branch again.
