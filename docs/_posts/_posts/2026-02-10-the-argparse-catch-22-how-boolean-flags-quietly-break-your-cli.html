Python’s argparse makes it trivially easy to add boolean flags.
Unfortunately, a small misunderstanding around store_true, store_false, and default can leave you with flags that:

  • parse correctly

  • appear in --help

  • pass tests

  • and yet do absolutely nothing

These bugs don’t crash your program. They quietly mislead users — which makes them far more dangerous than obvious failures.

This post explains why this happens, how to spot it, and exactly which combinations are valid or broken.

The mental model you must get right

Boolean flags in argparse are one-way setters, not toggles.

They only take effect when the flag is present.

store_true

  • flag present → value becomes True

  • flag absent → value defaults to False

store_false

  • flag present → value becomes False

  • flag absent → value defaults to True

Those defaults already exist.
Trouble starts when we override them incorrectly.

Safe, idiomatic patterns (use these)

Enable-only flags (default off)

parser.add_argument(
    "--verbose",
    action="store_true"
)

This is the most common and correct pattern.

Disable-only flags (default on)

parser.add_argument(
    "--no-cache",
    action="store_false",
    dest="use_cache"
)

Use this when the feature should be enabled by default.

The subtle mistake: overriding defaults

Many developers explicitly set default= for clarity — and accidentally create logical traps.

Redundant but harmless

parser.add_argument(
    "--debug",
    action="store_true",
    default=False
)

This works, but adds no value.
store_true already defaults to False.

Same story with:

parser.add_argument(
    "--no-auth",
    action="store_false",
    default=True
)

These aren’t bugs — just unnecessary noise.

The Catch-22: flags that cannot change anything

This is where things break.

store_true with default=True

parser.add_argument(
    "--feature",
    action="store_true",
    default=True
)

The flag exists.
The help text looks fine.
The behavior never changes.

You’ve created a Catch-22:

“You may enable this feature, but it is already enabled and cannot be disabled.”

store_false with default=False

parser.add_argument(
    "--no-logging",
    action="store_false",
    default=False
)

Same problem, inverted.

Why this causes real production bugs

These flags:

  • don’t raise errors

  • are often “tested” only on the default path

  • mislead automation scripts and CI pipelines

Someone passes a flag expecting behavior to change — and nothing happens.

That’s the worst kind of bug.

The only correct way to support both enable and disable

If you genuinely need symmetric control, use:

parser.add_argument(
    "--feature",
    action=argparse.BooleanOptionalAction,
    default=True
)

This creates:

  • --feature

  • --no-feature

Explicit, symmetric, and impossible to misuse.

How to use this table

A boolean flag is valid only if it can change the value in at least one direction.

If:

  • value when absent == value when present

then the flag must not exist.

Practical rules (pin these)

  1. Do not set default for store_true or store_false unless you fully understand the consequence

  2. Default False → use a positive flag (--enable-x)

  3. Default True → use a negative flag (--no-x)

  4. If a flag cannot change behavior, it is broken — even if argparse accepts it

  5. For symmetric enable/disable, use BooleanOptionalAction

One-line takeaway

If a CLI flag cannot change the value in at least one direction, you’ve built a Catch-22 — not a feature.
