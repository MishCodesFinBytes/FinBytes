Most confusion comes from mixing these two ideas. This diagram makes the separation explicit.

Diagram: What Scope Controls vs What Location Controls

┌──────────────────────────────────────────┐
│              PYTEST FIXTURE               │
└──────────────────────────────────────────┘
               │
               ▼
┌──────────────────────────┐
│  Scope (lifetime)        │
│  ─────────────────────  │
│  function / class        │
│  module / package        │
│  session                 │
│                          │
│  ➜ WHEN is it created?   │
│  ➜ WHEN is it destroyed? │
└──────────────────────────┘
               │
               │  (completely independent)
               ▼
┌──────────────────────────┐
│  Location (visibility)   │
│  ─────────────────────  │
│  inside class            │
│  module-level            │
│  conftest.py             │
│                          │
│  ➜ WHO can use it?       │
└──────────────────────────┘

Key Insight (call this out in the blog)

Scope controls lifetime.
Location controls visibility.
Placement does not imply sharing.

Concrete Illustration

test_example.py
│
├── @pytest.fixture          ← function scope (default)
│   def counter():           ← visible to entire file
│       return []
│
├── class TestA:
│   ├── test_one(counter)    ← new instance
│   └── test_two(counter)    ← new instance
│
└── class TestB:
    └── test_three(counter)  ← new instance

Even though the fixture is above the classes, it is not shared.

The CI-Only Failure (Realistic and Painful)

This is the kind of bug that:

  • passes locally

  • fails only in CI

  • wastes hours

The Intent (what the author thought)

“This fixture is function-scoped, so each test is isolated.”

The Code (looks innocent)

# test_cache.py

import pytest

shared_cache = {}

@pytest.fixture
def cache():
    return shared_cache

class TestUserCache:
    def test_add_user(self, cache):
        cache["alice"] = 1
        assert cache["alice"] == 1

    def test_cache_empty(self, cache):
        assert cache == {}

What Happens Locally

You run one test at a time:

pytest test_cache.py::TestUserCache::test_cache_empty

✅ Passes

pytest test_cache.py::TestUserCache::test_add_user

✅ Passes

What Happens in CI

CI runs the full suite:

pytest

Execution order (example):

test_add_user
test_cache_empty

Failure:

E   AssertionError: assert {'alice': 1} == {}

Why This Happens (the misunderstanding)

  • Fixture scope is function → fixture execution is isolated

  • But the fixture returns a module-level mutable object

  • The object identity is shared

Important distinction:

pytest isolates fixture calls, not objects you return

The Subtle Mental Trap

Many developers believe:

function scope → no shared state

But the real rule is:

function scope → fixture code re-runs
shared object → state still leaks

Why This Often Appears Only in CI

CI environments differ in critical ways:

This exposes hidden coupling that local runs mask.

The Correct Fix (Two Safe Options)

✅ Option 1: Create a fresh object inside the fixture

@pytest.fixture
def cache():
    return {}

Best default.

✅ Option 2: Reset shared state explicitly (if unavoidable)

@pytest.fixture
def cache():
    shared_cache.clear()
    return shared_cache

Only acceptable when:

  • Performance demands reuse

  • You explicitly control mutation

Add This Rule to the Blog (Very Shareable)

Fixture scope does not guarantee object isolation.
Only object creation does.

Bonus: How This Gets Even Worse with Wider Scope

@pytest.fixture(scope="session")
def cache():
    return {}

Now you’ve created:

  • global mutable state

  • shared across the entire test run

  • order-dependent failures

  • CI-only flakiness

This is one of the most common pytest anti-patterns.

Summary Section

  • Defining a fixture above a class does not change its scope

  • Default scope is always function

  • Fixture location controls visibility, not sharing

  • Mutable shared objects bypass scope guarantees

  • CI exposes what local runs hide
