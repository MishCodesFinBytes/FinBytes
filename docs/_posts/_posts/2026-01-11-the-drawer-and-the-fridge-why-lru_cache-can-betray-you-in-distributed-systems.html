The Drawer and the Fridge: Why lru_cache Can Betray You in Distributed Systems

Part 1: The problem — explained like to a child

Imagine a restaurant.

  • There are many kitchens (web servers)

  • Each kitchen has many cooks (workers)

  • Downstairs there is one big fridge everyone shares (Redis)

Each cook also has:

  • A small desk drawer next to them

That drawer is lru_cache.

What the drawer is good for

If a cook keeps:

  • A recipe

  • A rule

  • A math answer

in their drawer, that’s great.

They don’t need to run downstairs every time.

What the big fridge is for

The fridge stores:

  • Today’s food

  • Customer orders

  • Things that change

Everyone sees the same fridge.

The problem 😬

One cook puts apples in their drawer.

Later, someone replaces the apples in the fridge with bananas.

Now:

  • Cook A serves apples 🍎

  • Cook B serves bananas 🍌

Same question.
Different answers.

Customers get confused.

👉 This is exactly what happens when lru_cache is used for shared data.

Part 2: The same idea: a real life scenario

Each box is a server.
Each server has its own memory.
Redis sits in the middle.

🟢 Good code example

(Recipe in the drawer, food in the fridge)

from functools import lru_cache

# Drawer: a recipe (never changes)
@lru_cache(maxsize=128)
def format_price(amount):
    return f"£{amount:.2f}"


# Fridge: real data (can change)
def get_user_balance(user_id, redis):
    balance = redis.get(f"balance:{user_id}")
    return format_price(balance)

Why this is safe

  • format_price(10) is always the same

  • If Redis changes, formatting logic does not care

  • Being “a bit wrong” here causes no harm

🔴 Bad code example

(Food in the drawer — dangerous)

from functools import lru_cache

# ❌ BAD: shared data cached locally
@lru_cache(maxsize=1024)
def get_user_balance(user_id, redis):
    return redis.get(f"balance:{user_id}")

What goes wrong

  1. Balance changes in Redis

  2. lru_cache keeps the old value

  3. Other servers don’t know

  4. Users see inconsistent balances

This bug does not always show up in tests.
It shows up in production.

The bug in one line 🐞

Redis changed, but lru_cache didn’t hear about it.

Part 3: The grown-up explanation

Now let’s remove the story and say this properly.

What lru_cache actually is

lru_cache is:

  • In-process

  • Per-worker

  • Non-distributed

  • Memory-only

  • Unaware of other servers

In a typical enterprise setup:

Load Balancer
 ├─ Server A
 │   ├─ Worker 1 (lru_cache)
 │   └─ Worker 2 (lru_cache)
 ├─ Server B
 │   ├─ Worker 1 (lru_cache)
 │   └─ Worker 2 (lru_cache)
 └─ Redis (shared cache)

Each lru_cache is completely isolated.

Why this breaks correctness

Redis gives you:

  • Shared state

  • TTL

  • Invalidation

  • Consistency (eventual or strong, depending on use)

lru_cache gives you:

  • None of the above

So when you layer them incorrectly:

  • Redis invalidation does nothing

  • Pub/sub does nothing

  • Cache clears don’t propagate

  • Autoscaling creates cold, inconsistent caches

Memory amplification problem

If you have:

  • 10 servers

  • 8 workers each

  • lru_cache(maxsize=1000)

You now have 80,000 cached entries, silently.

This:

  • Increases RSS

  • Complicates capacity planning

  • Makes OOM kills harder to diagnose

The correct enterprise mental model

lru_cache must never be part of your correctness story.

When lru_cache is acceptable

✅ Pure functions
✅ Deterministic output
✅ Small key space
✅ No user or tenant context
✅ No freshness requirement

Examples:

  • Parsing schemas

  • Formatting values

  • Compiling regexes

  • Building SQL templates

When it is not acceptable

❌ User balances
❌ Permissions
❌ Feature flags
❌ Prices
❌ Anything invalidated elsewhere

If it would be bad to be wrong, even briefly:
don’t cache it locally.

Final rule to remember

lru_cache is a speed trick, not a cache strategy.

Or, from the child version:

🟢 Rules go in the drawer
🔴 Facts stay in the fridge
