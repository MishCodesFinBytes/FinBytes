We can turn Redis chat demo (prev post) into a full learning/troubleshooting exercise. I’ll lay it out step by step, including how to intentionally cause issues, what you’d see in logs, and how to troubleshoot them. We’ll structure it by stage of the system: app, Redis, caching, and multi-worker scenarios.

Project Setup

  • Python + Flask + Redis

  • Windows: Redis standalone or Waitress for multi-threaded simulation

  • Optional: Gunicorn (Linux/WSL/Docker) for multi-worker scenarios

1️⃣ Stage: App Layer – Cache Miss / TTL Expiration

How to Cause Issue

  • Request /history before sending any messages → Redis list is empty

  • Wait longer than TTL (default 5 min) after sending messages → cache expires

What You See in Logs

  • "Cache miss! Messages not found (expired or not populated)."

  • Redis console shows nothing because keys expired

How to Troubleshoot

  1. Verify TTL of key:

redis-cli TTL chat_messages

  • Returns -2 → key does not exist (expired)

  • Returns -1 → key exists but has no expiration

  1. Ensure messages are being stored with LPUSH and EXPIRE

  2. If key missing: check app logic or Redis connectivity

2️⃣ Stage: Redis Down / Connection Issues

How to Cause Issue

  • Stop redis-server.exe

  • Call /chat or /history

What You See in Logs

  • Python app raises connection error:

redis.exceptions.ConnectionError: Error 111 connecting to localhost:6379.

  • Flask endpoint returns 500 error

How to Troubleshoot

  1. Check if Redis is running:

redis-cli ping
# Should return PONG

  1. Restart Redis server

  2. If persistent issue: check firewall, port 6379 availability, or hostname

3️⃣ Stage: Pub/Sub – Message Not Received

How to Cause Issue

  • Start Flask app, but do not start listener thread (Chat.start_listener())

  • Send messages → Pub/Sub has no subscribers

What You See in Logs

  • Sender logs show status sent

  • No messages appear on console for any client

How to Troubleshoot

  1. Ensure listener thread is running (daemon=True)

  2. Check subscription:

pubsub = r.pubsub()
pubsub.subscribe("global_chat")
print(pubsub.channels)

  • Must include global_chat

4️⃣ Stage: Multi-Instance / Multi-Worker Race Conditions

How to Cause Issue

  • Run multiple Flask instances (or Gunicorn with multiple workers)

  • Send messages at the same time

  • Use /history endpoint

What You See in Logs

  • Some messages may appear out-of-order

  • If TTLs overlap, you may see missing or partial history

How to Troubleshoot

  1. Verify atomic operations: LPUSH + LTRIM are atomic in Redis

  2. Check for multi-worker issues:

    • Gunicorn logs: each worker writes separately

    • Race conditions can occur if multiple processes try to update a key without atomic commands

  3. Use transactions or Lua scripts if consistency is critical

5️⃣ Stage: Overloaded Cache / Eviction

How to Cause Issue

  • Reduce TTL (e.g., 5 sec)

  • Send >10 messages quickly

  • Keys start expiring automatically

What You See in Logs

  • /history shows fewer messages than expected

  • "Cache miss!" appears intermittently

How to Troubleshoot

  1. Check key length:

redis-cli LLEN chat_messages

  1. Check TTL:

redis-cli TTL chat_messages

  1. Adjust TTL or LTRIM logic to avoid premature eviction

6️⃣ Step-by-Step Demo Workflow

  1. Start Redis redis-server.exe

  2. Run chat app python chat_cache_demo.py

  3. Send a message

POST /chat {"message":"Hello"}

  1. Check history immediately → cache hit

GET /history

  1. Wait 5 minutes → cache expired → cache miss

  2. Stop Redis → try sending a message → connection error

  3. Start multiple listener threads → send messages → check race conditions

  4. Reduce TTL, send many messages → eviction logs and missing messages

7️⃣ Optional: Run with Waitress (Windows Production Simulation)

from waitress import serve
serve(app, host="0.0.0.0", port=5000)

  • Multiple threads simulate concurrent workers

  • Useful to demo race conditions, cache misses, TTL issues
