Both Redis and RabbitMQ can act as message brokers. The question is which one to reach for, and when using both makes sense.

Decision framework

Use Redis when

  • You already have Redis in the stack — zero extra infrastructure
  • Latency matters more than guaranteed delivery — Redis pub/sub is faster
  • Messages have a short useful life — latest price tick, cache invalidation signal
  • You need a simple task queue — Redis Lists + BRPOP is sufficient
  • You want Celery — it works on Redis, no RabbitMQ needed

Use RabbitMQ when

  • Messages represent work that must not be lost — order execution, trade confirmations
  • Consumers may be intermittently offline — RabbitMQ holds messages until they reconnect
  • You need complex routing — fanout to multiple consumers, topic-based routing
  • You need dead-letter queues — failed messages route to a DLQ automatically
  • You need per-message acknowledgement with visibility into unacked messages

Implementing a task queue in Redis (without Celery)

import redis
import json
import time

r = redis.Redis(decode_responses=True)

# Producer
def enqueue(task: dict) -> None:
    r.lpush('task_queue', json.dumps(task))

# Consumer
def worker() -> None:
    while True:
        # BRPOP blocks until a message arrives (0 = no timeout)
        _, raw = r.brpop('task_queue', timeout=0)
        task = json.loads(raw)
        try:
            process(task)
        except Exception as e:
            # Manual dead-letter: push to failed queue
            r.lpush('task_queue:failed', raw)
            print(f"Failed: {e}")

Using both together

A common pattern in trading systems: RabbitMQ for durable order flow (must not lose a message), Redis for ephemeral market data (latest price only, latency critical).

Dead-letter queues in RabbitMQ

Configure a DLQ so failed messages don't vanish:

channel.queue_declare(
    queue='tasks',
    durable=True,
    arguments={
        'x-dead-letter-exchange': 'dlx',
        'x-message-ttl': 60000,   # expire after 60s if unprocessed
    }
)

# Declare DLX and DLQ
channel.exchange_declare(exchange='dlx', exchange_type='direct')
channel.queue_declare(queue='tasks.failed', durable=True)
channel.queue_bind(queue='tasks.failed', exchange='dlx', routing_key='tasks')

# Now: nack with requeue=False -> routes to tasks.failed
ch.basic_nack(delivery_tag=method.delivery_tag, requeue=False)

Dead-letter queues are one of RabbitMQ's strongest features — they make failure handling explicit and inspectable. You can replay, inspect, or alert on the DLQ without any custom logic in your consumer.
