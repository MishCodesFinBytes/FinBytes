Redis-safe alternatives (what to use instead)

If the data can change and correctness matters, you need caching mechanisms that all servers agree on.

Here are the safe options, from simplest to most robust.

1️⃣ Redis with TTL (the boring but correct default)

This is the enterprise baseline.

Why it’s safe

  • Shared across all servers

  • Time-based expiry

  • Easy to reason about

  • Predictable failure modes

Example

def get_user_balance(user_id, redis, db):
    key = f"balance:{user_id}"

    balance = redis.get(key)
    if balance is not None:
        return balance

    balance = db.fetch_balance(user_id)
    redis.set(key, balance, ex=300)  # 5-minute TTL
    return balance

Use this when:

  • Data changes occasionally

  • Being a few minutes stale is acceptable

  • You want simplicity over cleverness

2️⃣ Redis + dogpile (stampede protection)

Problem:
When a cache expires, 100 requests hit the database at once.

Solution:
Let one worker refresh, others wait.

Pattern

def get_user_balance(user_id, redis, db):
    key = f"balance:{user_id}"
    lock_key = f"{key}:lock"

    value = redis.get(key)
    if value is not None:
        return value

    if redis.set(lock_key, "1", nx=True, ex=5):
        # I won the lock
        value = db.fetch_balance(user_id)
        redis.set(key, value, ex=300)
        redis.delete(lock_key)
        return value

    # Someone else is refreshing
    return redis.get(key)

Why enterprises like this

  • Prevents database meltdowns

  • Still fully Redis-coherent

  • Predictable under load

3️⃣ Client-side caching with Redis (RESP3)

Some Redis clients support local cache with server invalidation.

This gives you:

  • Local speed

  • Central correctness

Conceptually

Server Memory (local)
   ↑      ↓  invalidation
Redis (authoritative)

When Redis changes a key, it tells clients to drop it.

Pros

  • Very fast

  • Correct across servers

Cons

  • Client-specific

  • More complex to operate

  • Requires Redis ≥ 6 and proper client support

This is the only safe way to mix local memory + Redis for mutable data.

4️⃣ cachetools.TTLCache (single-process only, but honest)

If you are:

  • Single server

  • Or background jobs

  • Or batch pipelines

This is better than lru_cache because it has time expiry.

from cachetools import TTLCache

cache = TTLCache(maxsize=1000, ttl=300)

def get_exchange_rate(pair):
    if pair in cache:
        return cache[pair]

    rate = fetch_rate(pair)
    cache[pair] = rate
    return rate

⚠️ Still not multi-server safe
But at least:

  • It expires

  • It behaves predictably

5️⃣ Async refresh (“serve stale, refresh in background”)

Used heavily in high-traffic systems.

Pattern

  • Serve cached value (even if slightly stale)

  • Trigger background refresh

  • Never block users

def get_price(product_id, redis):
    key = f"price:{product_id}"
    value = redis.get(key)

    if value is None:
        enqueue_refresh(product_id)
        return "unknown"

    if redis.ttl(key) < 30:
        enqueue_refresh(product_id)

    return value

Why this works

  • Users stay fast

  • Data converges quickly

  • Redis remains the source of truth

🚫 What NOT to do (even though it’s tempting)

@lru_cache
def redis_get(key):
    return redis.get(key)

This:

  • Breaks invalidation

  • Breaks consistency

  • Breaks trust

If Redis is involved, do not hide it behind lru_cache.

The decision table (simple and honest)

Final takeaway

lru_cache optimises CPU work.
Redis manages shared truth.

When you mix them:

  • Redis must stay in charge

  • Local memory must stay disposable
