If you are comfortable with:

from unittest.mock import patch

…then pytest’s monkeypatch will feel both familiar and slightly unsettling.

It does less.
It looks simpler.
It has no decorators.
It has no context manager.
It auto-reverts.

And yet — it solves 90% of patching problems cleanly.

This guide explains monkeypatch in terms a unittest.mock.patch user already understands.

Mental Model Shift

unittest.mock.patch

  • Creates a mock object

  • Replaces a target with that mock

  • Tracks calls

  • Can autospec

  • Can assert calls

  • Requires decorator/context manager

pytest.monkeypatch

  • Reassigns an attribute

  • Does not create mocks for you

  • Does not track calls

  • Automatically restores state after the test

In short:

patch = mocking + patching
monkeypatch = patching only

Side-by-Side Comparison

Example Code

module2.py

class Class2:
    def multiply(x, y):
        return x * y

module1.py

from module2 import Class2

class Class1:
    def compute(self, a, b):
        return Class2.multiply(a, b) + 10

Using unittest.mock.patch

from unittest.mock import patch
from module1 import Class1

def test_with_patch():
    with patch("module1.Class2.multiply", return_value=100) as mock_mul:
        obj = Class1()
        result = obj.compute(3, 5)

        assert result == 110
        mock_mul.assert_called_once_with(3, 5)

You get:

  • automatic mock object

  • return_value

  • call assertions

Using monkeypatch

def test_with_monkeypatch(monkeypatch):
    monkeypatch.setattr("module1.Class2.multiply", lambda x, y: 100)

    obj = Class1()
    result = obj.compute(3, 5)

    assert result == 110

Notice what’s missing:

  • no mock object

  • no assert_called

  • no return_value kwarg

  • no context manager

It’s just reassignment.

How to Replicate return_value=...

patch(..., return_value=100) becomes:

monkeypatch.setattr("module1.Class2.multiply", lambda x, y: 100)

That’s it.

How to Replicate Call Assertions

If you need call tracking, you combine monkeypatch with Mock.

from unittest.mock import Mock

def test_call_tracking(monkeypatch):
    mock_mul = Mock(return_value=100)

    monkeypatch.setattr("module1.Class2.multiply", mock_mul)

    obj = Class1()
    result = obj.compute(3, 5)

    assert result == 110
    mock_mul.assert_called_once_with(3, 5)

Now you have both:

  • monkeypatch lifecycle management

  • full mock inspection

Instance Method Differences

If patching an instance method at class level:

monkeypatch.setattr(
    "module1.Class2.some_method",
    lambda self, x: 42
)

You must include self.

If patching a specific instance:

obj = Class1()

monkeypatch.setattr(
    obj.helper,
    "some_method",
    lambda x: 42
)

No self required (already bound).

Constructor Patching

Equivalent to:

with patch("module1.Class2") as MockClass:

Monkeypatch version:

from unittest.mock import Mock

def test_constructor(monkeypatch):
    fake_instance = Mock()
    fake_instance.multiply.return_value = 100

    monkeypatch.setattr("module1.Class2", lambda: fake_instance)

    obj = Class1()

You replace the class with a callable returning your fake.

Feature Comparison Table

Where Monkeypatch Is Actually Better

1️⃣ Environment Variables

monkeypatch.setenv("API_KEY", "test-key")

No need for patch.dict(os.environ, ...).

2️⃣ Dictionary Overrides

monkeypatch.setitem(config_dict, "timeout", 1)

3️⃣ Cleaner Test Flow

No indentation nesting.
No decorator stacking.
Less visual noise.

The Import Rule Still Applies

Exactly the same rule as patch:

Patch where the symbol is looked up.

If module1 imports Class2, patch:

module1.Class2

Never:

module2.Class2

This rule does not change between the two systems.

When To Use Which

Use unittest.mock.patch when:

  • You need autospec

  • You want heavy mock inspection

  • You rely heavily on call assertions

Use monkeypatch when:

  • You want simpler tests

  • You’re overriding deterministic behavior

  • You’re patching environment/config

  • You prefer flat test structure

In practice, many mature codebases combine both.

The Big Realization

patch feels powerful because it bundles:

  • replacement

  • mock creation

  • lifecycle

  • introspection

monkeypatch is intentionally minimal.

It modifies attributes.
It restores them.
It gets out of your way.

Once you separate “mocking” from “patching” conceptually, monkeypatch becomes predictable and elegant.
