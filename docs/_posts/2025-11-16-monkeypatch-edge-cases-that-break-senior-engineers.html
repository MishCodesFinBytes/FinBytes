Monkeypatch is simple.

It reassigns attributes and restores them after the test.

And yet — even experienced engineers regularly lose hours to subtle edge cases.

This post covers the real-world scenarios that cause silent failures, flaky CI builds, and “why is this not patched?” debugging sessions.

1️⃣ Patching the Wrong Import Location

The Classic Failure

# module1.py
from module2 import Class2

def run():
    return Class2.do_work()

You patch:

monkeypatch.setattr("module2.Class2.do_work", lambda: 42)

Test still calls the original function.

Why?

Because module1 already holds its own reference to Class2.

You must patch:

monkeypatch.setattr("module1.Class2.do_work", lambda: 42)

Diagnosis Tip

Print the object identity:

import module1, module2
print(module1.Class2 is module2.Class2)

If it prints True, you’re safe.
If not — you’re patching the wrong namespace.

2️⃣ Patching After the Object Was Bound

If the dependency is stored in __init__, patching too late won’t work.

class Service:
    def __init__(self):
        self.client = ExternalClient()

If you do:

obj = Service()
monkeypatch.setattr("module.ExternalClient", FakeClient)

Too late. The instance already exists.

Correct Approach

Patch first:

monkeypatch.setattr("module.ExternalClient", FakeClient)
obj = Service()

Order matters.

3️⃣ Forgetting self on Instance Methods

This is subtle.

monkeypatch.setattr(
    "module.Class.method",
    lambda x: 10   # WRONG
)

You get:

TypeError: takes 1 positional argument but 2 were given

Because Python injects self.

Correct version:

lambda self, x: 10

If patching a specific instance:

monkeypatch.setattr(obj, "method", lambda x: 10)

Now self is already bound — so don’t include it.

4️⃣ Staticmethod vs Classmethod Confusion

If the original method is:

class C:
    @classmethod
    def foo(cls, x):
        ...

Your patch must accept cls:

lambda cls, x: 10

If it’s a @staticmethod, no self or cls.

Mismatch here causes misleading TypeErrors.

5️⃣ Late Imports Inside Functions

def run():
    from module2 import helper
    return helper()

You patch:

monkeypatch.setattr("module2.helper", lambda: 42)

This works — but only if patching occurs before run() is called.

However, if run() was imported elsewhere before patching, behavior can differ depending on execution order.

Late imports complicate patch reasoning.

6️⃣ Patching Builtins Incorrectly

Example:

open("file.txt")

Wrong:

monkeypatch.setattr("builtins.open", fake_open)

Correct — but only if the module uses open directly.

If the module does:

from builtins import open

You must patch:

module_name.open

Again: patch where looked up.

7️⃣ Mutable Defaults and Shared State

Monkeypatch restores attributes —
but it does not reset mutated objects.

Example:

CONFIG = {"timeout": 10}

If your code mutates CONFIG, monkeypatch won’t protect you.

You must:

monkeypatch.setitem(CONFIG, "timeout", 1)

Or copy before use.

This is a common CI-only flake source.

8️⃣ Patching Properties

Properties aren’t plain attributes.

class User:
    @property
    def name(self):
        return "real"

Naive patch:

monkeypatch.setattr("module.User.name", lambda self: "fake")

Won’t behave as expected.

Correct approach:

monkeypatch.setattr(
    "module.User.name",
    property(lambda self: "fake")
)

Properties must remain descriptors.

9️⃣ Async Functions

If the original function is async:

async def fetch():
    ...

Patching with a normal lambda:

lambda: 42

Will break because it isn’t awaitable.

Correct:

async def fake():
    return 42

or

monkeypatch.setattr("module.fetch", AsyncMock(return_value=42))

🔟 Silent Success, Wrong Behavior

The most dangerous case:

You patch correctly.
The test passes.
But the real dependency was never used in that test path.

You think you tested failure handling — you didn’t.

Mitigation strategy:

Combine monkeypatch with Mock:

mock = Mock(return_value=42)
monkeypatch.setattr("module.func", mock)

...
mock.assert_called_once()

Trust, but verify.

1️⃣1️⃣ CI-Only Failures

Why does monkeypatch work locally but fail in CI?

Common causes:

  • Different import order

  • Different working directory

  • Implicit relative imports

  • Module reloading via coverage tools

  • Shared global state between tests

CI exposes weak isolation.

1️⃣2️⃣ Reloading Modules (Advanced)

If a module caches values at import time:

VALUE = expensive_call()

Patching expensive_call later won’t affect VALUE.

You may need:

import importlib
import module
importlib.reload(module)

This is rare — but brutal when encountered.

Failure Pattern Summary

Senior-Level Debug Strategy

When a patch “doesn’t work,” ask:

  1. Where is the symbol resolved?

  2. Was it bound earlier?

  3. Is it staticmethod/classmethod/property?

  4. Is it async?

  5. Am I mutating shared state instead?

  6. Did the code path even execute?

Systematic diagnosis beats guessing.

The Real Takeaway

Monkeypatch is simple.

The Python object model is not.

Most failures aren’t about pytest —
they’re about:

  • namespace resolution

  • descriptor behavior

  • object binding

  • import timing

Once you understand those, monkeypatch becomes entirely predictable.
