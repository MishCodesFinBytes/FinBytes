A Silent CI Failure Caused by Import Cache Pollution

This post describes a class of failure we encountered in a large enterprise Python environment.

The key lesson is about how subtle test code can destabilise shared CI systems, especially when that code is generated by AI and not fully personally checked by the author.

The Incident

A small change was merged into a service that owned its own unit tests.
All local tests passed.
The PR was approved.

Within days:

  • Central CI builds began failing intermittently

  • Failures appeared in unrelated services

  • Errors were non-deterministic

  • Rollbacks did not consistently fix the issue

The build team initially suspected:

  • Infrastructure flakiness

  • Dependency resolution issues

  • CI node instability

The actual cause was none of these.

The Root Cause (Simplified)

A pytest fixture mutated sys.modules to mock a dependency:

sys.modules["some_dependency"] = Mock()

The fixture:

  • Was not function-scoped, and was above the test class definition

  • Passed local tests

  • Appeared harmless

But in a shared CI environment, this mutation:

  • Persisted longer than expected

  • Interacted with coverage tooling

  • Affected unrelated imports

  • Leaked across test boundaries - and caused unrelated tests to randomly fail - which could not be replicated on a local dev machine.

Why This Took Days to Find

1. The Failure Was Silent

Mocks don’t raise errors.
They absorb them.

Instead of crashes, the system produced:

  • empty configs

  • skipped logic

  • partially initialised components

2. Failures Appeared Elsewhere

The code that caused the issue:

  • belonged to one service

  • but broke downstream build steps - because the mocked module was a generic one like sys module for example

  • and affected other teams’ pipelines

No direct error pointed back to the fixture.

Why This Is Common in Enterprise CI

Enterprise CI systems often have:

  • Long-lived Python processes

  • Coverage instrumentation

  • Parallel test execution

  • Shared artifacts

  • Multiple stages after tests

All of these increase sensitivity to global interpreter state.

The AI Factor

The fixture was generated using an AI coding assistant.

The AI:

  • Solved the immediate testing problem

  • Used a technically valid approach

  • Did not warn about global state

  • Had no understanding of CI topology

This is not an AI bug.
It’s a human responsibility gap.

AI can generate working code that is operationally unsafe.

Checklist: Reviewing AI-Generated Test Code

This checklist can be used to check AI-generated code, before using it.

1. Global State Checks

Ask explicitly:

  • Does this touch sys, os.environ, sys.modules, or globals?

  • Is the mutation fully reversible?

  • Who cleans it up?

If the answer isn’t obvious → reject or rewrite.

2. Import-Level Mocking

Red flags:

  • sys.modules[...] = ...

  • mocking standard libraries

  • mocking config loaders

Required:

  • pytest.monkeypatch

  • function-scoped fixtures only

3. Fixture Scope Review

AI often defaults to:

@pytest.fixture
def ...

Verify:

  • Not autouse

  • Not module or session scoped

  • Not defined in conftest.py unless intentional

4. CI Awareness Questions

Before approving:

  • Would this behave differently under pytest --cov?

  • Would test order matter?

  • Would parallel execution change the outcome?

If unsure → assume yes.

5. Failure Mode Analysis

Ask:

  • If this fails, will it fail loudly?

  • Or will it return empty / mocked data?

Silent failures are worse than crashes.

6. Ownership and Blast Radius

  • Who owns this fixture?

  • Who else could be affected?

  • Could another team’s code import this dependency?

Enterprise code has shared consequences.

Example time:

Wrong Example (Leaky Fixture)

# conftest.py
import pytest

@pytest.fixture(scope="module")
def counter():
    return {"value": 0}






# test_one.py
class TestOne:
    def test_increment(self, counter):
        counter["value"] += 1
        assert counter["value"] == 1






# test_two.py
class TestTwo:
    def test_starts_at_zero(self, counter):
        assert counter["value"] == 0

Why this is wrong

  • counter is module-scoped

  • It returns a mutable object

  • State leaks between tests

  • Test order determines success or failure

Local run:

pytest test_two.py

✅ Passes

CI run:

pytest

❌ Fails (counter already incremented)

Correct Example (Isolated Fixture)

# conftest.py
import pytest

@pytest.fixture(scope="function")
def counter():
    return {"value": 0}






# test_one.py
class TestOne:
    def test_increment(self, counter):
        counter["value"] += 1
        assert counter["value"] == 1






# test_two.py
class TestTwo:
    def test_starts_at_zero(self, counter):
        assert counter["value"] == 0

Why this works

  • Each test gets a fresh dictionary

  • No shared state

  • Tests pass:

    • individually

    • together

    • in any order

    • in parallel

One-Sentence Rule

If a pytest fixture returns a mutable object, it should almost always be function scoped.

Optional Extra (One-line Safety Guard)

@pytest.fixture
def counter():
    return {"value": 0}.copy()

A Simple Defensive CI Test - to detect leaky module mocks (which is the most commonly occuring issue)

This is a low-cost guardrail:

def test_no_mocked_modules_leaked():
    import sys
    from unittest.mock import Mock
    leaked = [
        name for name, mod in sys.modules.items()
        if isinstance(mod, Mock)
    ]
    assert not leaked, f"Mocked modules leaked into sys.modules: {leaked}"

This would have caught the issue immediately.

Final Takeaway

Passing tests are not proof of correctness in enterprise environments.
Test code that mutates global state can destabilise entire build systems — silently.

AI makes it faster to write tests.
It does not make them safe by default.

In shared CI systems, test code deserves the same scrutiny as production code.
