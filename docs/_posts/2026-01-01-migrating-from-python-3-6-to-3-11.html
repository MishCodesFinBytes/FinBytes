A Practical Enterprise Guide

Upgrading Python in enterprise systems is rarely about syntax.
The challenge is implicit assumptions that worked in Python 3.6 but break in 3.11 — often silently or only in CI.

This guide summarises key differences, why they matter, and how to catch them before they cause production failures.

1. Dates and Datetimes

  • Python 3.11 is stricter when mixing date and datetime.

  • Legacy code often did implicit coercion that 3.6 tolerated.

from datetime import date, datetime

d = date(2024, 1, 1)
dt = datetime(2024, 1, 1, 0, 0)

dt - d  # TypeError in Python 3.11

Fix: Convert explicitly:

dt - datetime.combine(d, datetime.min.time())

Tip: Also convert Pandas columns with pd.to_datetime.

2. Ordering Changes

  • Sets: always unordered; 3.6 sometimes appeared stable.

  • Dict iteration: insertion order guaranteed in 3.7+, but implicit order assumptions still fail.

Example failure:

assert list(set(["a", "b", "c"])) == ["a", "b", "c"]  # May fail in 3.11

Fix: Compare sorted() lists or sets explicitly:

assert sorted(list(set(...))) == ["a", "b", "c"]

3. Truthiness and None Comparisons

  • Ambiguous truthiness (if df["column"]:) now raises errors in Python 3.11.

  • None comparisons with < or > now reliably raise TypeError.

Fix:

if not df["column"].empty:
if x is not None and y is not None and x < y:

4. Exception Messages

  • Python 3.11 improved error reporting.

  • Tests relying on exact exception messages may fail.

Fix: Assert on exception type, not text:

with pytest.raises(ValueError):
    int("abc")

5. Floats and Decimal Behaviour

  • Exact float equality sometimes fails due to precision and rounding changes.

  • Python 3.11 makes numeric behaviour more visible.

Fix: Use tolerances:

import math
assert math.isclose(result, expected, rel_tol=1e-9)

6. String and Bytes Handling

  • Python 3.11 exposes implicit string/bytes mixing errors earlier.

  • Often happens in logging, serialization, or messaging pipelines.

Fix: Explicitly encode/decode:

b"abc" + "def".encode()

7. Performance-Exposed Timing Issues

  • Python 3.11 is faster.

  • Tests relying on time.sleep() or implicit ordering may fail.

Fix: Poll for conditions or use proper synchronisation instead of time-based sleeps.

8. Warnings

  • Deprecated or future warnings may now fail CI when run with -W error.

  • Python 3.11 surfaces them more aggressively.

Fix: Resolve warnings; do not ignore or suppress.

9. is vs ==

  • Python 3.6 sometimes masked is used for value equality.

  • Python 3.11 exposes this reliably.

Fix:

if status == "OK":

3.6 → 3.11 Migration Checklist

Optional CI Enforcement for Upgrade Safety

  • Run tests under both Python 3.6 and 3.11.

  • Fail on:

    • Mixing date and datetime

    • Ambiguous truthiness

    • None comparisons in numeric operators

    • Exact exception message assertions

  • Example:

pytest -W error --strict-markers
python ci_checks/check_datetime_none.py

Final Takeaway

Python 3.11 doesn’t break your code — it exposes assumptions that 3.6 silently tolerated.

Upgrading isn’t about syntax fixes; it’s about making implicit behaviour explicit, safe, and deterministic.
Following this checklist and enforcing it in CI can save days of debugging and unstable builds in enterprise systems.
