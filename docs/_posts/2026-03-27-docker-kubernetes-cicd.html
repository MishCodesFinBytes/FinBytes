Three tools that show up in almost every production engineering job description. You don't need to be a DevOps engineer — but you do need to know what they are, why they exist, and how to work with them when your code runs in a container, gets tested in a pipeline, and deploys to a cluster.

Part 1 — Docker

The problem it solves

"It works on my machine" is a real, career-affecting problem. Your Python service runs on 3.11 with specific packages. The production server runs 3.9 with slightly different versions. A dependency installs differently on macOS vs Linux. These aren't hypotheticals — they cause production incidents.

Docker solves this by packaging your application together with its entire environment: the OS base, the runtime, the libraries, the config. That package (an image) runs identically everywhere — your laptop, a CI server, a production cluster.

Container vs virtual machine

A VM emulates an entire computer, including its own OS kernel. Heavy, slow to start, large on disk. A container shares the host OS kernel but isolates the filesystem, processes, and network. Lightweight, starts in milliseconds, small on disk. For most application workloads, containers are the right tool.

Image vs container

An image is the blueprint — a read-only snapshot of your application and its environment. A container is a running instance of an image. One image can run as many containers as you like. Think of an image as a class definition and a container as an instance of that class.

The Dockerfile

A Dockerfile is a recipe for building an image. Each instruction adds a layer. Layers are cached — if a layer hasn't changed, Docker reuses it. This makes rebuilds fast.

# Start from an official Python base image
FROM python:3.11-slim

# Set working directory inside the container
WORKDIR /app

# Copy requirements first (cached separately — only reruns when requirements change)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy the rest of the application
COPY . .

# Expose the port the app listens on
EXPOSE 8000

# Command to run when the container starts
CMD ["python", "-m", "uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
Copy requirements.txt and install dependencies before copying your source code. Docker caches each layer — if you copy everything first, any code change invalidates the pip install cache and reinstalls all packages every build. Order matters.

Volumes — persisting data

Containers are ephemeral — when they stop, any data written inside them is lost. Volumes solve this by mounting a directory from the host (or a managed volume) into the container.

# Mount current directory into /app inside the container
docker run -v $(pwd):/app myapp:latest

# Named volume (Docker manages the location)
docker run -v mydata:/var/lib/postgresql/data postgres:15

docker-compose — running multiple containers

Most real applications aren't a single container. A typical setup: your app, a database, a cache. docker-compose defines and runs multi-container applications from a single YAML file.

# docker-compose.yml
version: "3.9"

services:
  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      - REDIS_URL=redis://redis:6379
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
    depends_on:
      - redis
      - db
    volumes:
      - .:/app   # mount source for live reload in development

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

  db:
    image: postgres:15
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
docker-compose up          # start everything
docker-compose up -d       # start in background
docker-compose down        # stop and remove containers
docker-compose logs -f app # follow logs for the app service
docker-compose exec app bash  # shell into the app container
In the compose file, services refer to each other by service name. Your app connects to Redis at redis:6379 not localhost:6379 — because inside the Docker network, redis is the hostname. This trips people up the first time.

Part 2 — Kubernetes

What problem k8s solves that Docker alone doesn't

Docker runs containers on one machine. When you need to run containers across many machines — for scale, redundancy, or rolling deployments — you need an orchestrator. Kubernetes (k8s) is that orchestrator. It handles: scheduling containers onto nodes, restarting crashed containers, scaling up/down, rolling deployments with zero downtime, service discovery, and secrets management.

As a developer you won't manage the cluster itself — that's platform/ops territory. But you will write or edit the YAML files that describe how your application runs, and you'll use kubectl to inspect and debug your running services.

Core concepts

Pod — the smallest deployable unit. Usually one container, sometimes a main container plus a sidecar (e.g. a logging agent). Pods are ephemeral — they come and go. You never interact with pods directly in normal operation.

Deployment — declares how many replicas of your pod should run, which image to use, and how to roll out updates. If a pod crashes, the Deployment restarts it. If you update the image, the Deployment does a rolling update.

Service — a stable network endpoint in front of your pods. Since pods come and go (and their IP addresses change), a Service provides a fixed address. Types: ClusterIP (internal only), LoadBalancer (exposes externally), NodePort (exposes on a port on each node).

ConfigMap — key/value config injected into pods as environment variables or mounted files. Non-sensitive config: feature flags, URLs, timeouts.

Secret — like ConfigMap but base64-encoded and access-controlled. Database passwords, API keys, TLS certificates.

Namespace — logical grouping of resources. Common pattern: dev, staging, prod namespaces in one cluster.

A real Deployment YAML

apiVersion: apps/v1
kind: Deployment
metadata:
  name: pricer
  namespace: trading
  labels:
    app: pricer
spec:
  replicas: 3                    # run 3 copies
  selector:
    matchLabels:
      app: pricer
  template:
    metadata:
      labels:
        app: pricer
    spec:
      containers:
        - name: pricer
          image: myrepo/pricer:v1.4.2    # always use a specific tag, not :latest
          ports:
            - containerPort: 8000
          env:
            - name: REDIS_URL
              value: "redis://redis-service:6379"
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: pricer-secrets   # Secret object name
                  key: db-password       # key inside the Secret
          resources:
            requests:
              memory: "128Mi"
              cpu: "100m"
            limits:
              memory: "512Mi"
              cpu: "500m"
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8000
            initialDelaySeconds: 10
            periodSeconds: 30
          readinessProbe:
            httpGet:
              path: /readyz
              port: 8000
            initialDelaySeconds: 5
            periodSeconds: 10

A Service YAML

apiVersion: v1
kind: Service
metadata:
  name: pricer-service
  namespace: trading
spec:
  selector:
    app: pricer           # routes to pods with this label
  ports:
    - protocol: TCP
      port: 80            # port the Service listens on
      targetPort: 8000    # port on the pod
  type: ClusterIP         # internal only — use LoadBalancer to expose externally

Never use :latest as the image tag in a Deployment. It makes rollbacks impossible and creates non-deterministic deployments. Always use a specific version tag — a semantic version, a git SHA, or a build number.
Things you don't need to know yet: Ingress controllers, StatefulSets, DaemonSets, Operators, Helm chart authoring, cluster provisioning, RBAC configuration. If someone asks you about these before you've shipped anything to k8s, they're interviewing for an SRE role, not a developer role.

Part 3 — CI/CD with GitHub Actions

What CI/CD actually means

Continuous Integration (CI): every code push automatically runs tests, linting, and builds. You find out immediately if you broke something, before it reaches anyone else.

Continuous Deployment (CD): after CI passes, the code is automatically deployed to an environment. Fully automated = continuous deployment. Requires human approval = continuous delivery.

The pipeline stages for a typical Python service:

push code
    → lint (ruff, flake8)
    → type check (mypy)
    → unit tests (pytest)
    → integration tests
    → build Docker image
    → push image to registry
    → deploy to staging
    → (manual approval)
    → deploy to production

GitHub Actions anatomy

A workflow is a YAML file in .github/workflows/. It contains one or more jobs. Each job runs on a runner (a fresh VM) and contains a sequence of steps. Steps run shell commands or call pre-built actions from the marketplace.

# .github/workflows/ci.yml

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt
          pip install -r requirements-dev.txt

      - name: Lint
        run: ruff check .

      - name: Type check
        run: mypy src/

      - name: Run tests
        run: pytest --tb=short -q

Building and pushing a Docker image in CI

  build-and-push:
    runs-on: ubuntu-latest
    needs: test              # only runs if test job passes
    if: github.ref == 'refs/heads/main'   # only on main branch

    steps:
      - uses: actions/checkout@v4

      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            myrepo/pricer:latest
            myrepo/pricer:${{ github.sha }}

Secrets in CI — never hardcode

Add secrets in your GitHub repo under Settings → Secrets and variables → Actions. Reference them in workflows as ${{ secrets.MY_SECRET }}. They're masked in logs — you'll see *** if they're accidentally printed.

Never put passwords, API keys, or tokens in your workflow YAML file or your source code. GitHub will actually scan for common secret patterns and alert you, but don't rely on that. Environment variables from secrets is the only acceptable approach.

Other CI platforms — same ideas, different syntax

If you work somewhere that uses Jenkins, GitLab CI, Azure DevOps, or CircleCI — the concepts are identical. You have triggers, stages, steps, and artifact outputs. The YAML syntax differs but everything you just learned transfers directly. GitHub Actions is the easiest to start with because the runner infrastructure is fully managed and the marketplace has actions for almost everything.

How it all connects

Here's the end-to-end flow for a containerised service:

1. You push code to main
       ↓
2. GitHub Actions triggers
   - runs pytest
   - runs mypy
   - if all pass: builds Docker image tagged with git SHA
   - pushes image to Docker Hub / ECR / GCR
       ↓
3. CD step updates the Deployment YAML with the new image tag
   kubectl set image deployment/pricer pricer=myrepo/pricer:abc1234
       ↓
4. Kubernetes does a rolling update
   - starts new pods with the new image
   - waits for readiness probe to pass
   - terminates old pods
   - if new pods fail readiness: automatic rollback
       ↓
5. Your change is live with zero downtime

What you own as a developer

  • Dockerfile — how your app is packaged
  • docker-compose.yml — local development environment
  • .github/workflows/ci.yml — the CI pipeline
  • k8s/deployment.yaml — how your app runs in the cluster
  • The health check endpoints (/healthz, /readyz) your liveness and readiness probes call

What ops / platform owns

  • The cluster itself — nodes, networking, storage
  • Ingress controllers, TLS termination
  • Cluster-level RBAC and security policies
  • The container registry
  • Monitoring infrastructure (Prometheus, Grafana, alerting)
The single most valuable thing you can do before your first k8s deployment: implement proper health check endpoints. A /healthz that returns 200 when the app is alive, and a /readyz that returns 200 only when the app is ready to serve traffic (feed connected, cache warm, etc.). Without these, k8s can't do rolling deployments safely. With them, it handles failures automatically.
