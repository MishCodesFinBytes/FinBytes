You have a library function that sends emails.

It’s called inside ClassA.

You want to test:

  • How many times it was called

  • The order of calls

  • The exact to, cc, from, subject, and body used

And you want to do it properly — not by guessing, not by printing, not by over-mocking.

This guide walks through:

  1. Using monkeypatch + Mock (pytest-native style)

  2. Using unittest.mock.patch

  3. How mock.call fits into both

  4. Clean patterns for senior-level tests

Example Setup

email_lib.py

def send_email(to, cc, sender, subject, body):
    # Real implementation sends email
    pass

class_a.py

from email_lib import send_email


class ClassA:

    def process(self):
        send_email(
            to=["user1@test.com"],
            cc=[],
            sender="noreply@test.com",
            subject="Report 1",
            body="First report"
        )

        send_email(
            to=["user2@test.com"],
            cc=["manager@test.com"],
            sender="noreply@test.com",
            subject="Report 2",
            body="Second report"
        )

🔑 Critical Rule (Before We Continue)

Always patch where the function is used, not where it is defined.

You must patch:

class_a.send_email

NOT:

email_lib.send_email

Because ClassA imported it directly.

If you patch the wrong namespace, the test may pass while your patch does nothing.

Part 1 — Using mock.call with monkeypatch

monkeypatch does not create mocks.

You supply the mock yourself.

Test Using Monkeypatch + Mock

from unittest.mock import Mock, call
from class_a import ClassA


def test_email_calls(monkeypatch):

    # Create a mock manually
    email_mock = Mock(name="send_email_mock")

    # Patch where ClassA looks up send_email
    monkeypatch.setattr("class_a.send_email", email_mock)

    obj = ClassA()
    obj.process()

    # 1️⃣ Verify call count
    assert email_mock.call_count == 2

    # 2️⃣ Verify order and exact arguments
    expected_calls = [
        call(
            to=["user1@test.com"],
            cc=[],
            sender="noreply@test.com",
            subject="Report 1",
            body="First report"
        ),
        call(
            to=["user2@test.com"],
            cc=["manager@test.com"],
            sender="noreply@test.com",
            subject="Report 2",
            body="Second report"
        )
    ]

    assert email_mock.call_args_list == expected_calls

Understanding mock.call

call(...) constructs a structured representation of a function call.

Internally, each call captures:

  • Positional args

  • Keyword args

  • Order of invocation

email_mock.call_args_list is a list of call(...) objects in execution order.

That is what enables exact verification.

Cleaner Assertion Pattern

Instead of manually comparing lists:

email_mock.assert_has_calls(expected_calls, any_order=False)

This:

  • Validates order

  • Produces clearer failure messages

Inspecting Individual Calls

Sometimes you don’t want strict matching.

You can inspect directly:

first_call = email_mock.call_args_list[0]

assert first_call.kwargs["subject"] == "Report 1"
assert first_call.kwargs["to"] == ["user1@test.com"]

This is useful when:

  • Body contains dynamic content

  • You only care about certain fields

Part 2 — Same Test Using unittest.mock.patch

Now the equivalent using patch.

from unittest.mock import patch, call
from class_a import ClassA


def test_email_calls_patch():

    with patch("class_a.send_email") as email_mock:
        obj = ClassA()
        obj.process()

        expected_calls = [
            call(
                to=["user1@test.com"],
                cc=[],
                sender="noreply@test.com",
                subject="Report 1",
                body="First report"
            ),
            call(
                to=["user2@test.com"],
                cc=["manager@test.com"],
                sender="noreply@test.com",
                subject="Report 2",
                body="Second report"
            )
        ]

        email_mock.assert_has_calls(expected_calls, any_order=False)

Differences:

  • patch automatically creates the mock

  • Requires context manager or decorator

  • Otherwise identical in verification logic

Advanced Pattern: Partial Matching

If subject must match but body is dynamic:

from unittest.mock import ANY

email_mock.assert_has_calls([
    call(
        to=["user1@test.com"],
        cc=[],
        sender="noreply@test.com",
        subject="Report 1",
        body=ANY
    )
])

ANY ignores that field.

Advanced Pattern: Preventing Duplicate Emails

assert email_mock.call_count == 1

Simple but powerful guard against accidental double-send.

Senior-Level Testing Strategy

When testing side effects like email:

  1. Verify number of calls

  2. Verify order

  3. Verify key arguments

  4. Use ANY for non-deterministic fields

  5. Fail loudly on mismatch

Do not just assert call_count.
That misses argument integrity.

Key Takeaway

To use mock.call in a monkeypatch scenario:

  1. Create a Mock()

  2. Patch the dependency with that mock

  3. Use call(...) to define expected invocations

  4. Assert via call_args_list or assert_has_calls

Monkeypatch handles lifecycle.
Mock handles introspection.

Together, they give you full observability of side effects.
