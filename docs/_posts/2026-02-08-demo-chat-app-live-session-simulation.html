Here’s a visual flow diagram for the Redis chat demo showing how a message flows and where issues can happen. This is great for live teaching, borrowed from blog.bytebytego.com

Diagram Explanation

Flow:

  1. Client sends message → /chat endpoint on Flask

  2. Flask processes request

    • Publishes message to Redis Pub/Sub channel → for real-time listeners

    • Adds message to Redis cache list with TTL (LPUSH + LTRIM + EXPIRE)

  3. Listener threads / other app instances

    • Subscribed to Pub/Sub → receive messages live

    • New listeners joining later may miss messages if relying only on Pub/Sub

  4. Client retrieves history → /history

    • Cache hit → messages returned from Redis list

    • Cache miss → list empty (TTL expired or not populated)

  5. Potential Issues / Troubleshooting Points

    • Cache Miss → TTL expired / key not populated

    • Pub/Sub missed messages → listener not started

    • Redis Down → connection errors

    • Multi-worker race → simultaneous writes without atomic pipeline (solved by LPUSH+LTRIM pipeline)

    • Cache Eviction → MAX_MESSAGES limit triggers removal of old messages

Troubleshooting:

Here’s a step-by-step demo script you can use to show all the Redis cache and Pub/Sub issues live with your multi-worker chat app. It’s structured like a 10–15 minute live session, with each step showing the issue, expected console logs, and discussion points.

Goal: Demonstrate cache hits/misses, TTL expiration, Pub/Sub delivery, race conditions, and Redis connection issues.

Setup Before Demo

  1. Start Redis:

redis-server.exe

  1. Run Python app:

python chat_multi_worker_demo.py

  1. Confirm Flask running at http://127.0.0.1:5000/

  2. Console shows:

[HH:MM:SS] Listener1 started listening...
[HH:MM:SS] Listener2 started listening...
[HH:MM:SS] Starting multi-worker chat demo...

Step 1: Show Cache Miss (0:00–1:00)

Action:

  • Open /history endpoint in browser/Postman:

GET http://127.0.0.1:5000/history

Expected Logs:

[HH:MM:SS] [CACHE MISS] No messages in cache

Talking Points:

  • Cache miss happens because no messages yet

  • TTL is irrelevant at this point

Step 2: Send a Message (1:00–2:00)

Action:

POST http://127.0.0.1:5000/chat
Body: {"message":"Hello World!"}

Expected Logs:

[HH:MM:SS] [PUB/SUB SENT] Hello World!
[HH:MM:SS] [CACHE] Message cached (TTL=10s)
[HH:MM:SS] Listener1 received: Hello World!
[HH:MM:SS] Listener2 received: Hello World!

Talking Points:

  • Pub/Sub delivers to all subscribers

  • Message also stored in Redis list for history

Step 3: Show Cache Hit (2:00–3:00)

Action:

  • Call /history again:

GET http://127.0.0.1:5000/history

Expected Logs:

[HH:MM:SS] [CACHE HIT] Retrieved 1 messages

Talking Points:

  • Messages retrieved from Redis cache instead of re-computing

  • Demonstrates benefit of caching

Step 4: Demonstrate Cache Expiration (3:00–5:00)

Action:

  • Wait 10–15 seconds (CACHE_TTL = 10s)

  • Call /history again

Expected Logs:

[HH:MM:SS] [CACHE MISS] No messages in cache

Talking Points:

  • TTL expiration automatically removes old messages

  • Shows importance of setting appropriate TTL

Step 5: Show Redis Down / Connection Error (5:00–6:00)

Action:

  • Stop Redis server

  • Send a message:

POST /chat {"message":"Test after Redis down"}

Expected Logs:

[HH:MM:SS] [ERROR] Redis not available! Message not sent.

Talking Points:

  • Connection error handling in app

  • Discuss troubleshooting steps: restart Redis, check port

Step 6: Demonstrate Multi-Worker Race Conditions (6:00–10:00)

Action:

  1. Start multiple instances of the app (or use Waitress with multiple threads)

from waitress import serve
serve(app, host="0.0.0.0", port=5000)

  1. Send multiple messages quickly:

POST /chat {"message":"Message 1"}
POST /chat {"message":"Message 2"}
...
POST /chat {"message":"Message 5"}

Expected Logs:

  • Multiple listener threads receive messages

[HH:MM:SS] Listener1 received: Message 1
[HH:MM:SS] Listener2 received: Message 1
[HH:MM:SS] Listener1 received: Message 2
[HH:MM:SS] Listener2 received: Message 2
...

  • Console shows all PUB/SUB and cache logs

Talking Points:

  • Demonstrates atomic operations via pipeline prevent partial updates

  • Discuss race conditions, what happens if LPUSH+LTRIM not atomic

  • Discuss TTL: overlapping expirations in multiple workers

Step 7: Show Eviction / TTL Overwrites (10:00–12:00)

Action:

  • Keep sending messages rapidly (> MAX_MESSAGES = 10)

  • Check /history after 10s

Expected Logs:

  • Only last 10 messages retrieved

[HH:MM:SS] [CACHE HIT] Retrieved 10 messages

  • Older messages discarded automatically

Talking Points:

  • Shows eviction strategy

  • TTL + LTRIM controls cache size

  • Helps explain real-world cache issues with limits

Step 8: Wrap-Up (12:00–15:00)

Discussion Points:

  • Cache misses, hits, TTL expiration

  • Pub/Sub message delivery and missed subscribers

  • Redis down scenarios & error handling

  • Multi-worker race conditions & atomic operations

  • Cache eviction strategies (max items + TTL)

Optional Extensions for Live Demo:

  • Adjust TTL to 5s → show faster expiration

  • Stop/start Redis mid-demo → live errors

  • Open multiple console tabs → simulate multiple users

Quick Tips for Live Demo

  1. Adjust TTL to 5–15s for fast observation of expiration.

  2. MAX_MESSAGES = 10 → easy to show eviction.

  3. Use multiple console windows → simulate multiple workers / listeners.

  4. Stop/start Redis → show connection errors and recovery.

  5. Explain atomic pipeline → LPUSH + LTRIM prevents partial writes.

  6. Highlight logs clearly → [CACHE HIT], [CACHE MISS], [PUB/SUB SENT], [ListenerX received].
