--- layout: post title: "RabbitMQ — what it is and how it differs from Redis pub/sub" date: 2026-05-27 tags: [rabbitmq, queues, message-queue, misc] section: queues permalink: /misc/rabbitmq-what-is-it/ ---

Redis pub/sub is fire-and-forget. If a subscriber isn't connected when you publish, the message is gone. RabbitMQ is different: it persists messages in queues, routes them by rules, and guarantees delivery even if the consumer is offline.

Core concepts

Producer — sends messages to an exchange, not directly to a queue.

Exchange — routes messages to queues based on rules (bindings). Types: direct, fanout, topic, headers.

Queue — buffer that stores messages until a consumer processes them.

Consumer — receives and acknowledges messages. Until it acknowledges, the message stays in the queue and can be redelivered.

Binding — rule connecting an exchange to a queue, optionally with a routing key.

vs Redis pub/sub

FeatureRedis pub/subRabbitMQ
Message persistenceNo — fire and forgetYes — durable queues survive restarts
Consumer offlineMessage lostMessage held until consumer reconnects
AcknowledgementNoneConsumer acks; redelivered on failure
RoutingChannel name onlyExchange types, routing keys, bindings
ThroughputVery high (in-memory)High, slightly lower (persistence overhead)
SetupZero — already using RedisSeparate broker to deploy

When to use which

Use Redis pub/sub when: you need very low latency, losing occasional messages is acceptable, and you're already using Redis. Good for: live market data ticks, UI notifications, cache invalidation signals.

Use RabbitMQ when: you need guaranteed delivery, consumers may be offline, messages represent work that must not be lost. Good for: order execution, trade confirmations, EOD batch jobs.

Exchange types

  • Direct — routes to queues where routing key = binding key exactly. One-to-one.
  • Fanout — broadcasts to all bound queues. Ignores routing key. Use for pub/sub.
  • Topic — routing key patterns with * (one word) and # (zero or more words). prices.*.equity matches prices.london.equity.
  • Headers — routes on message header attributes rather than routing key.
