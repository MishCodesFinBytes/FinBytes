Why SOLID matters in quant code

SOLID is not abstract theory. It is a set of heuristics that make code easier to test, extend, and reason about. In a pricing system, violations cluster in predictable places: a class that prices and persists and logs (SRP), a pricer that needs modifying every time a new model is added (OCP), or a risk engine that takes a concrete database rather than an interface (DIP).

  • S - Single Responsibility: a class should have one reason to change
  • O - Open/Closed: open for extension, closed for modification
  • L - Liskov Substitution: subtypes must be substitutable for their base types
  • I - Interface Segregation: clients should not depend on interfaces they do not use
  • D - Dependency Inversion: depend on abstractions, not concretions

All examples use a simplified options pricing system. The same Pricer evolves through each principle to show before and after.

Python SOLID examples

S - Single Responsibility

# BAD: one class, three reasons to change
class BadPricer:
    def price(self, trade): ...
    def save_to_db(self, trade): ...
    def log_result(self, result): ...

# GOOD: separated concerns
class Pricer:         def price(self, trade) -> float: ...
class TradeRepository:def save(self, trade) -> None: ...
class PricingLogger:  def log(self, result) -> None: ...

O - Open/Closed

from abc import ABC, abstractmethod

class PricingModel(ABC):
    @abstractmethod
    def price(self, trade) -> float: ...

class BlackScholes(PricingModel):
    def price(self, trade) -> float: ...

class MonteCarlo(PricingModel):
    def price(self, trade) -> float: ...

# Add Binomial without touching existing code
class Binomial(PricingModel):
    def price(self, trade) -> float: ...

D - Dependency Inversion

from typing import Protocol

class PositionStore(Protocol):
    def get_positions(self, portfolio_id: str) -> list: ...

class RiskEngine:
    def __init__(self, store: PositionStore):
        self.store = store  # injected: testable, swappable

C# SOLID examples

S - Single Responsibility

public class Pricer          { public double Price(Trade t) { ... } }
public class TradeRepository { public void Save(Trade t)   { ... } }
public class PricingLogger   { public void Log(double r)   { ... } }

O - Open/Closed

public interface IPricingModel { double Price(Trade trade); }

public class BlackScholes : IPricingModel { public double Price(Trade t) { ... } }
public class MonteCarlo   : IPricingModel { public double Price(Trade t) { ... } }

public class Pricer {
    private readonly IPricingModel _model;
    public Pricer(IPricingModel model) => _model = model;
    public double Price(Trade t) => _model.Price(t);
}

I - Interface Segregation

// BAD: fat interface forces implementors to stub unused methods
public interface IBadFeed {
    double GetSpot(string sym);
    double GetVol(string sym);
    OrderBook GetBook(string sym);  // not all consumers need this
}

// GOOD: split into focused interfaces
public interface ISpotFeed { double GetSpot(string sym); }
public interface IVolFeed  { double GetVol(string sym);  }

public class Pricer {
    public Pricer(ISpotFeed spots, IVolFeed vols) { ... }
}

L - Liskov Substitution

// BAD: subtype throws where base does not
public class VolSurface {
    public virtual double GetVol(double strike, double expiry) => 0.2;
}
public class EmptyVolSurface : VolSurface {
    public override double GetVol(double strike, double expiry)
        => throw new NotSupportedException();  // VIOLATION
}

// GOOD: return nullable instead
public double? TryGetVol(double strike, double expiry) => null;

C++ SOLID examples

S - Single Responsibility

class Pricer          { public: double price(const Trade& t); };
class TradeRepository { public: void   save(const Trade& t);  };
class PricingLogger   { public: void   log(double result);    };

O - Open/Closed: runtime polymorphism

struct IPricingModel {
    virtual ~IPricingModel() = default;
    virtual double price(const Trade& t) const = 0;
};

struct BlackScholes : IPricingModel {
    double price(const Trade& t) const override { return 0; }
};

class Pricer {
    std::unique_ptr<IPricingModel> model_;
public:
    explicit Pricer(std::unique_ptr<IPricingModel> m) : model_(std::move(m)) {}
    double price(const Trade& t) const { return model_->price(t); }
};

O - Open/Closed: compile-time (zero-overhead)

template<typename Model>
class StaticPricer {
    Model model_;
public:
    double price(const Trade& t) const { return model_.price(t); }
};

StaticPricer<BlackScholes> bs_pricer;
StaticPricer<MonteCarlo>   mc_pricer;

D - Dependency Inversion

struct IPositionStore {
    virtual ~IPositionStore() = default;
    virtual std::vector<Position> get_positions(const std::string& id) const = 0;
};

class RiskEngine {
    const IPositionStore& store_;
public:
    explicit RiskEngine(const IPositionStore& store) : store_(store) {}
};
C++ gives you two open/closed strategies: runtime polymorphism (virtual functions, testable, flexible) and compile-time polymorphism (templates, zero overhead). Use runtime when the model is chosen at runtime. Use templates when it is fixed at compile time and performance matters.
