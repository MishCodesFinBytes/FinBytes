The at-least-once problem

In any distributed system, message delivery is either at-most-once (may lose messages) or at-least-once (may deliver duplicates). At-least-once is usually safer - better to process twice than not at all - but only if your processing is idempotent.

Idempotency: an operation is idempotent if applying it multiple times has the same effect as applying it once. Setting a position to 100 is idempotent. Adding 100 to a position is not.

Finance examples: a fill message delivered twice must not double the position. A trade confirmation processed after a system restart must be recognised as already processed. An EOD job that fails halfway and restarts must not produce duplicate records.

Solution - idempotency keys: assign a unique ID to each operation. Before processing, check if the ID has been seen. If yes, return the cached result. Store processed IDs in a persistent store. Use database unique constraints as enforcement.

Python idempotency

import hashlib, json

class IdempotentTradeProcessor:
    def __init__(self): self._processed: dict[str, dict] = {}

    def process_fill(self, fill: dict) -> dict:
        key = fill.get("fill_id") or self._hash_fill(fill)
        if key in self._processed:
            return {**self._processed[key], "idempotent": True}
        result = self._apply_fill(fill)
        self._processed[key] = result
        return result

    def _apply_fill(self, fill: dict) -> dict:
        symbol   = fill["symbol"]
        quantity = fill["quantity"]   # absolute, not delta
        self._positions[symbol] = quantity  # SET not ADD
        return {"symbol": symbol, "quantity": quantity, "status": "processed"}

    def _hash_fill(self, fill: dict) -> str:
        canonical = json.dumps(fill, sort_keys=True).encode()
        return hashlib.sha256(canonical).hexdigest()[:16]

C# idempotent processing

public class IdempotentTradeProcessor {
    private readonly HashSet<string> _processedIds = new();
    private readonly Dictionary<string, double> _positions = new();

    public ProcessResult ProcessFill(Fill fill) {
        if (_processedIds.Contains(fill.FillId))
            return new ProcessResult(fill.Symbol, _positions.GetValueOrDefault(fill.Symbol), true);
        _positions[fill.Symbol] = fill.Quantity;  // SET not ADD
        _processedIds.Add(fill.FillId);
        return new ProcessResult(fill.Symbol, fill.Quantity, false);
    }
}

C++ idempotent processing

#include <string>
#include <unordered_set>
#include <unordered_map>

struct Fill { std::string fill_id, symbol; double quantity; };
struct ProcessResult { std::string symbol; double quantity; bool was_duplicate; };

class IdempotentFillProcessor {
    std::unordered_set<std::string>        processed_ids_;  // use persistent store in production
    std::unordered_map<std::string, double> positions_;
public:
    ProcessResult process(const Fill& fill) {
        if (processed_ids_.count(fill.fill_id))
            return {fill.symbol, positions_[fill.symbol], true};
        positions_[fill.symbol] = fill.quantity;   // SET not ADD
        processed_ids_.insert(fill.fill_id);
        return {fill.symbol, fill.quantity, false};
    }
};
Design your data model for idempotency from the start. Store absolute state (position = 100) rather than deltas (position += 10). Require idempotency keys on all mutating operations. Use database unique constraints as enforcement - they are the only reliable deduplication mechanism when multiple processes can run concurrently.
