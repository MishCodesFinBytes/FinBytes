How C++ finds names

When you write sort(v.begin(), v.end()), how does the compiler find sort? Name lookup in C++ is surprisingly complex — two-phase lookup for templates, argument-dependent lookup (ADL) for free functions, and namespace rules that can cause subtle bugs.

Argument-Dependent Lookup (ADL)

When you call a function with arguments from a namespace, the compiler also searches that namespace for the function name. This is why std::cout << obj finds your operator<< defined in the same namespace as obj — even without a using directive.

Why using namespace std is dangerous

It pulls all of std into the current scope. If std later adds a function whose name matches something in your code, compilation breaks. In header files, it poisons every file that includes it. The fix: use std:: prefix, or targeted using std::cout.

Two-phase lookup

Template bodies are parsed in two phases. Names not dependent on template parameters are resolved at definition time; dependent names are resolved at instantiation time. This is why some template errors appear only when you use the template, not when you define it.

Python name resolution — LEGB

# Python name resolution: Local, Enclosing, Global, Built-in
x = "global"

def outer():
    x = "enclosing"
    def inner():
        # x = "local"  # uncomment to shadow enclosing
        print(x)  # finds "enclosing" via LEGB
    inner()

outer()  # enclosing

# Import namespaces
import math
math.sqrt(4)  # explicit namespace — always clear

from math import sqrt  # import name into local scope
sqrt(4)                # clear, but now sqrt could shadow a local

from math import *    # imports everything — avoid this (like using namespace std)

C# — using directives and aliases

// using directive — imports namespace
using System.Collections.Generic;
var list = new List<int>();  // found via using

// Alias — avoids ambiguity
using Dict = System.Collections.Generic.Dictionary<string, double>;
var d = new Dict();

// Global using (C# 10+) — project-wide
// In a single file: global using System.Linq;

// Explicit namespace — always unambiguous
var list2 = new System.Collections.Generic.List<int>();

// No equivalent of ADL — extension methods are the closest
// "str".ToUpper() finds ToUpper via the string type, not namespace lookup

C++ name lookup in detail

ADL in action

namespace finance {
    struct Price { double value; };

    // ADL finds this when called with a finance::Price argument
    std::ostream& operator<<(std::ostream& os, const Price& p) {
        return os << "£" << p.value;
    }

    void swap(Price& a, Price& b) {  // ADL finds this for std::swap(a, b)
        std::swap(a.value, b.value);
    }
}

int main() {
    finance::Price p1{100.0}, p2{200.0};
    std::cout << p1 << "
";  // ADL finds finance::operator<<
    swap(p1, p2);               // ADL finds finance::swap (without std:: prefix!)
    return 0;
}

The using namespace std hazard

// count.h — NEVER do this in a header
using namespace std;  // poisons every file that includes count.h

// This creates ambiguity if the user also has their own count()
// or if std adds a new count() — both are real bugs in production code

// CORRECT: explicit prefix
std::vector<int> v = {1, 2, 3};
std::cout << std::count(v.begin(), v.end(), 2) << "
";

// ACCEPTABLE: targeted using in .cpp file (not header)
// using std::cout;
// using std::vector;

Two-phase lookup

template<typename T>
void process(T value) {
    helper(value);  // dependent name — resolved at instantiation
    // If T is from namespace foo, ADL will find foo::helper
}

// If you want a specific helper regardless of T:
template<typename T>
void process_explicit(T value) {
    ::helper(value);  // :: forces global scope lookup
}
💡 The practical takeaway: always use std:: prefix (or targeted using declarations in .cpp files). Never put using namespace std in a header. Understand ADL so you can write good operator<< and swap overloads for your types.
