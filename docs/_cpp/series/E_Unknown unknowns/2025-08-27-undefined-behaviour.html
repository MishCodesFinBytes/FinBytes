What undefined behaviour actually means

In Python and C#, wrong code throws an exception. In C++, undefined behaviour (UB) means the C++ standard places no requirement on what the program does. It might crash, produce wrong answers, appear to work, corrupt memory silently, or behave differently between debug and release builds. The compiler is free to assume UB never occurs — and optimise accordingly.

The most common sources

  • Signed integer overflow (unlike unsigned, which wraps)
  • Null pointer dereference
  • Out-of-bounds array access
  • Use-after-free (accessing memory after it's been deleted)
  • Data races (two threads, shared memory, at least one write, no sync)
  • Uninitialized variables

How to find it

Sanitisers are your best friend. Enable them during development:

  • -fsanitize=undefined — catches UB at runtime (UBSan)
  • -fsanitize=address — catches memory errors (ASan)
  • -fsanitize=thread — catches data races (TSan)

These add ~2x overhead — use in tests, not in production builds.

Python — exceptions, not silent corruption

# Python raises exceptions for most "undefined" operations

# Integer overflow: Python ints are arbitrary precision
x = 2**63        # works fine, no overflow
print(x)         # 9223372036854775808

# Index out of bounds: IndexError
lst = [1, 2, 3]
try:
    val = lst[10]
except IndexError as e:
    print(f"Caught: {e}")  # list index out of range

# None dereference: AttributeError
obj = None
try:
    obj.method()
except AttributeError as e:
    print(f"Caught: {e}")

# Use after "free": Python GC handles it
x = [1, 2, 3]
y = x          # reference, not copy
del x          # x deleted, but y still holds a reference
print(y)       # [1, 2, 3] — still valid
Python's safety comes at a cost — every operation has type checking, bounds checking, and reference counting overhead. C++ UB is the price you pay for skipping all of that.

C# — checked arithmetic and managed memory

// Integer overflow: wraps silently by default
int x = int.MaxValue;
int y = x + 1;
Console.WriteLine(y);  // -2147483648 — silent wrap!

// Use checked to make overflow throw
try {
    checked {
        int z = int.MaxValue + 1;  // OverflowException
    }
} catch (OverflowException e) {
    Console.WriteLine($"Caught: {e.Message}");
}

// Array bounds: IndexOutOfRangeException
int[] arr = { 1, 2, 3 };
try {
    var v = arr[10];
} catch (IndexOutOfRangeException e) {
    Console.WriteLine($"Caught: {e.Message}");
}

// Null: NullReferenceException (or use null-safety operators)
string? s = null;
Console.WriteLine(s?.Length);    // null (safe navigation)
Console.WriteLine(s ?? "empty"); // "empty" (null coalescing)

C++ — UB examples and how to catch them

Signed integer overflow (UB)

#include <climits>
#include <iostream>

int main() {
    int x = INT_MAX;
    int y = x + 1;  // UNDEFINED BEHAVIOUR
    // In optimised builds, the compiler may assume this never happens
    // and eliminate entire code paths that "can't" execute

    // Safe: use unsigned (wraps defined) or check first
    if (x == INT_MAX) {
        std::cout << "would overflow
";
    } else {
        int y2 = x + 1;
        std::cout << y2 << "
";
    }
    return 0;
}

Out-of-bounds access (UB)

#include <vector>

int main() {
    std::vector<double> v = {1.0, 2.0, 3.0};

    // UB — no bounds check in operator[]
    double bad = v[10];  // reads garbage memory, may crash, may not

    // Safe — throws std::out_of_range
    double safe = v.at(10);  // always checks

    // Safe — check first
    if (10 < v.size())
        double ok = v[10];

    return 0;
}

Use-after-free (UB)

#include <memory>
#include <iostream>

int main() {
    // Raw pointer — easy to get wrong
    int* p = new int(42);
    delete p;
    // std::cout << *p;  // UB — use after free

    // Smart pointer — safe
    auto sp = std::make_unique<int>(42);
    std::cout << *sp << "
";  // 42
    sp.reset();  // freed
    // std::cout << *sp;  // would throw/crash — sp is null
    if (sp) std::cout << *sp << "
";  // safe check

    return 0;
}

Enabling sanitisers (CMakeLists.txt)

option(ENABLE_SANITIZERS "Enable UBSan and ASan" OFF)

if(ENABLE_SANITIZERS)
    target_compile_options(myapp PRIVATE
        -fsanitize=address,undefined
        -fno-omit-frame-pointer
    )
    target_link_options(myapp PRIVATE
        -fsanitize=address,undefined
    )
endif()

# Build with: cmake -DENABLE_SANITIZERS=ON ..
💡 Always run your test suite with sanitisers enabled. UB that "works" in debug mode due to coincidental memory layout will bite you in production with optimisations enabled. ASan + UBSan together catch the vast majority of memory and arithmetic bugs.
