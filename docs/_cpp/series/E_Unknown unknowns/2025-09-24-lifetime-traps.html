Lifetimes — the hardest part of C++

A dangling reference is a reference to memory that has been freed. Accessing it is undefined behaviour. Unlike use-after-free with raw pointers (usually obvious), dangling references often look completely innocent — they're often caused by temporaries and lifetime extension rules that most developers don't know by heart.

Common traps for Python/C# developers

  • Returning a reference to a local variable
  • Storing a reference to a vector element, then modifying the vector
  • string_view outliving the string it refers to
  • Range-for over a temporary that's destroyed mid-loop
  • Structured bindings to temporaries

Detection

ASan (address sanitiser) catches most use-after-free and dangling references at runtime. -Wall -Wextra catches return-of-local-reference at compile time. Clang-tidy's lifetime checks catch more complex cases.

Python — references can't dangle (GC prevents it)

def get_list():
    local = [1, 2, 3]
    return local  # fine: returns reference, GC keeps the list alive

result = get_list()
print(result)  # [1, 2, 3] — no dangling

# Python objects stay alive as long as any reference to them exists
x = [1, 2, 3]
y = x           # y holds a reference
del x           # x deleted, but y still alive
print(y)        # [1, 2, 3] — fine

# string_view equivalent: Python strings are immutable, no slicing hazard
s = "Hello, world"
sub = s[7:]     # new string object — no reference to original storage
Python's GC guarantees objects live as long as any reference to them exists. C++ has no such guarantee — you manage lifetimes manually.

C# — GC prevents most lifetime issues

// Reference types: GC keeps them alive while referenced
int[] GetArray() {
    var local = new[] { 1, 2, 3 };
    return local;  // safe: GC keeps array alive
}

// Value types in unsafe context: can dangle (rare)
// C# ref local to managed heap is safe
ref int GetRef(int[] arr) => ref arr[0];  // safe while arr is alive

// Span<T>: CAN dangle if source is stack-allocated
Span<int> DangerousMethod() {
    Span<int> local = stackalloc int[3] { 1, 2, 3 };
    // return local;  // compile error! C# prevents this
    return Array.Empty<int>();  // must return heap data
}

// ReadOnlySpan<char> from string is safe — strings are heap-allocated
ReadOnlySpan<char> GetSpan(string s) => s.AsSpan()[7..];  // safe

C++ lifetime traps

Returning reference to local (compile warning)

// WRONG — returns reference to local that is destroyed on return
const double& get_price() {
    double price = 100.0;
    return price;  // compiler warning: reference to local
}  // price destroyed here

// CORRECT
double get_price_safe() { return 100.0; }  // return by value

Iterator/reference invalidation (silent UB)

#include <vector>

void bad_iteration() {
    std::vector<double> v = {1.0, 2.0, 3.0};
    const double& ref = v[0];   // reference to first element
    v.push_back(4.0);           // may reallocate!
    // std::cout << ref;        // UB — ref may be dangling
}

// Safe: use indices, not references, across mutations
void safe_iteration() {
    std::vector<double> v = {1.0, 2.0, 3.0};
    v.push_back(4.0);
    std::cout << v[0] << "
";  // always valid
}

string_view outliving the string

#include <string>
#include <string_view>

std::string_view get_symbol() {
    std::string s = "AAPL";
    return s;         // WRONG: s is destroyed, view dangles
}

std::string_view get_literal() {
    return "AAPL";    // OK: string literal has static lifetime
}

class Trade {
public:
    // WRONG: storing view of temporary
    void set_symbol(std::string_view sym) {
        symbol_ = sym;  // if sym views a temporary, it dangles
    }

    // CORRECT: store a string, not a view
    void set_symbol_safe(std::string_view sym) {
        symbol_str_ = std::string(sym);
    }
private:
    std::string_view symbol_;      // dangerous if not careful
    std::string      symbol_str_;  // always safe
};

Range-for over temporary

#include <vector>
#include <string>

std::vector<std::string> get_symbols() {
    return {"AAPL", "MSFT", "GOOGL"};
}

// SAFE: range-for binds the temporary's lifetime
for (const auto& sym : get_symbols()) {
    // sym is valid here
}

// DANGER: storing a reference to the temporary (C++23 fixed some cases)
// const auto& symbols = get_symbols();  // lifetime extended — actually safe
// for (const auto& sym : symbols) { }   // safe
💡 Enable -Wall -Wextra -Wlifetime (Clang) and run with ASan during tests. These catch most lifetime bugs at compile time or test time rather than in production.
