Two kinds of health

Liveness: is the process alive? Fails if the process is stuck in an infinite loop, deadlocked, or OOM. Correct response: restart the process.

Readiness: is the process ready to serve traffic? Fails if alive but warming up, draining, or dependencies unavailable. Correct response: stop routing traffic to it, but do not restart it.

Kubernetes uses both. A failed liveness probe triggers a restart. A failed readiness probe removes the pod from the service's endpoint list. For a pricing service: liveness = heartbeat still ticking; readiness = market data feed connected and vol surface loaded.

Silent failures: the hardest to detect. The process is running, the heartbeat is ticking, but it is processing incorrect data. For example: the feed is connected but returning stale prices. The process appears healthy but computes wrong Greeks. Staleness checks are essential.

Python health check server

from flask import Flask, jsonify
import threading, time

app    = Flask(__name__)
state  = {"ready": False, "feed_age_s": 0.0}
last_heartbeat = time.monotonic()

def processing_thread():
    global last_heartbeat
    while True:
        process_some_work()
        last_heartbeat = time.monotonic()
        time.sleep(0.1)

@app.get("/healthz")     # liveness
def liveness():
    age = time.monotonic() - last_heartbeat
    if age > 30.0:
        return jsonify({"status": "unhealthy", "reason": f"no heartbeat for {age:.0f}s"}), 503
    return jsonify({"status": "ok"})

@app.get("/readyz")      # readiness
def readiness():
    checks = {
        "feed_connected":     market_feed.is_connected(),
        "vol_surface_loaded": vol_surface is not None,
        "feed_fresh":         state["feed_age_s"] < 60.0,
    }
    if all(checks.values()):
        return jsonify({"status": "ready", "checks": checks})
    return jsonify({"status": "not_ready", "checks": checks}), 503

C# health checks (ASP.NET)

builder.Services.AddHealthChecks()
    .AddCheck("market_feed", () => {
        var age = DateTime.UtcNow - lastFeedUpdate;

            ? HealthCheckResult.Healthy($"Feed age: {age.TotalSeconds:F0}s")

            ? HealthCheckResult.Healthy()
            : HealthCheckResult.Unhealthy("Vol surface not loaded"));

app.MapHealthChecks("/healthz");  // liveness
app.MapHealthChecks("/readyz", new HealthCheckOptions {
    Predicate = check => check.Tags.Contains("ready")
});

C++ health endpoint + watchdog

#include <atomic>
#include <chrono>
#include <string>

std::atomic<int64_t> heartbeat_ns{0};
std::atomic<int64_t> last_feed_ns{0};

void update_heartbeat() {
    heartbeat_ns.store(
        std::chrono::steady_clock::now().time_since_epoch().count(),
        std::memory_order_relaxed);
}

bool is_alive(std::chrono::seconds max_age = std::chrono::seconds(30)) {
    auto now  = std::chrono::steady_clock::now().time_since_epoch().count();
    auto last = heartbeat_ns.load(std::memory_order_relaxed);
    return (now - last) < std::chrono::nanoseconds(max_age).count();
}

bool is_feed_fresh(std::chrono::seconds max_age = std::chrono::seconds(60)) {
    auto now  = std::chrono::steady_clock::now().time_since_epoch().count();
    auto last = last_feed_ns.load(std::memory_order_relaxed);
    return (now - last) < std::chrono::nanoseconds(max_age).count();
}

struct HealthStatus { bool live; bool ready; std::string message; };

HealthStatus get_health() {
    bool live  = is_alive();
    bool ready = live && is_feed_fresh() && vol_surface_loaded.load();
    std::string msg = ready ? "ok" : !live ? "deadlocked" :
                      !is_feed_fresh() ? "feed stale" : "not ready";
    return {live, ready, msg};
}
Separate liveness and readiness endpoints. Kubernetes calls them on different schedules with different thresholds. A liveness failure triggers a restart - make sure liveness only fails for genuinely unrecoverable conditions, not transient dependency issues.
