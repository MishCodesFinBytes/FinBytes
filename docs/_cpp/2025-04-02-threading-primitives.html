Threads in C++

C++11 introduced a portable threading library — std::thread, std::mutex, std::condition_variable. Before that, you used platform-specific APIs (pthreads on Linux, Win32 on Windows). Now there's one standard interface everywhere.

Data races

Two threads accessing the same memory location, at least one writing, with no synchronisation — undefined behaviour. Not "might be wrong" — the C++ standard says the program has no defined behaviour at all. Mutexes prevent this.

RAII locks

std::lock_guard and std::unique_lock acquire a mutex on construction and release it on destruction. They compose with exceptions — the mutex is always released even if the protected code throws. Never call mutex.lock()/mutex.unlock() directly.

condition_variable

For producer/consumer patterns: the consumer waits on a condition variable until the producer signals that work is available. More efficient than spinning.

Python threading — the GIL

Python's Global Interpreter Lock (GIL) means only one thread runs Python bytecode at a time. Threading in Python is mainly useful for I/O-bound work. For CPU-bound parallelism, use multiprocessing or concurrent.futures.ProcessPoolExecutor.

import threading
import time
from collections import deque

# Shared queue with lock
class SafeQueue:
    def __init__(self):
        self._q    = deque()
        self._lock = threading.Lock()
        self._cond = threading.Condition(self._lock)

    def push(self, item):
        with self._cond:
            self._q.append(item)
            self._cond.notify()

    def pop(self, timeout=1.0):
        with self._cond:
            self._cond.wait_for(lambda: len(self._q) > 0, timeout=timeout)
            return self._q.popleft() if self._q else None

q = SafeQueue()

def producer():
    for i in range(5):
        time.sleep(0.1)
        q.push(i)
        print(f"produced {i}")

def consumer():
    for _ in range(5):
        item = q.pop()
        print(f"consumed {item}")

t1 = threading.Thread(target=producer)
t2 = threading.Thread(target=consumer)
t1.start(); t2.start()
t1.join();  t2.join()

C# threading — Task and async/await

C# has both low-level Thread and high-level Task/async. For CPU-bound work, use Task.Run or Parallel. For shared state, use lock (syntactic sugar for Monitor).

using System.Collections.Concurrent;

// Thread-safe queue built in
var queue = new BlockingCollection<int>(boundedCapacity: 100);

// Producer
var producer = Task.Run(() => {
    for (int i = 0; i < 5; i++) {
        Thread.Sleep(100);
        queue.Add(i);
        Console.WriteLine($"produced {i}");
    }
    queue.CompleteAdding();
});

// Consumer
var consumer = Task.Run(() => {
    foreach (var item in queue.GetConsumingEnumerable())
        Console.WriteLine($"consumed {item}");
});

await Task.WhenAll(producer, consumer);

// Manual lock example
object lockObj = new();
int shared = 0;

Parallel.For(0, 1000, _ => {
    lock (lockObj) { shared++; }  // RAII-style
});
Console.WriteLine(shared);  // 1000

C++ threading primitives

Basic thread

#include <thread>
#include <iostream>

void worker(int id) {
    std::cout << "thread " << id << " running\n";
}

int main() {
    std::thread t1(worker, 1);
    std::thread t2(worker, 2);
    t1.join();  // wait for t1 to finish
    t2.join();  // wait for t2 to finish
    return 0;
}

Mutex and lock_guard

#include <thread>
#include <mutex>
#include <vector>
#include <iostream>

std::mutex mtx;
int shared_counter = 0;

void increment(int n) {
    for (int i = 0; i < n; ++i) {
        std::lock_guard<std::mutex> lock(mtx);  // acquired here
        ++shared_counter;
    }  // released here — even if exception thrown
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 8; ++i)
        threads.emplace_back(increment, 10'000);
    for (auto& t : threads) t.join();
    std::cout << shared_counter << "\n";  // always 80000
    return 0;
}

Producer/consumer with condition_variable

#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>
#include <optional>
#include <iostream>

template<typename T>
class SafeQueue {
public:
    void push(T item) {
        {
            std::lock_guard<std::mutex> lock(mtx_);
            q_.push(std::move(item));
        }
        cv_.notify_one();
    }

    std::optional<T> pop() {
        std::unique_lock<std::mutex> lock(mtx_);
        cv_.wait(lock, [&] { return !q_.empty() || done_; });
        if (q_.empty()) return std::nullopt;
        T item = std::move(q_.front());
        q_.pop();
        return item;
    }

    void done() {
        std::lock_guard<std::mutex> lock(mtx_);
        done_ = true;
        cv_.notify_all();
    }

private:
    std::queue<T>           q_;
    std::mutex              mtx_;
    std::condition_variable cv_;
    bool                    done_ = false;
};

int main() {
    SafeQueue<int> q;

    std::thread producer([&] {
        for (int i = 0; i < 5; ++i) {
            std::this_thread::sleep_for(std::chrono::milliseconds(50));
            q.push(i);
            std::cout << "produced " << i << "\n";
        }
        q.done();
    });

    std::thread consumer([&] {
        while (auto item = q.pop())
            std::cout << "consumed " << *item << "\n";
    });

    producer.join();
    consumer.join();
    return 0;
}
💡 Always use lock_guard or unique_lock — never call mutex.lock() directly. If an exception fires between lock() and unlock(), the mutex is never released and every subsequent thread deadlocks.
