Python · 7 yrs C# · 11 yrs C++ · day 1

I've been writing code professionally for nearly two decades. Eleven years in C# — enterprise software, financial systems, the whole stack. Then Python pulled me in, and for the last seven years that's been my daily language: data pipelines, quant models, automation, analysis.

Both are managed languages. Both handle memory for you. Both let you think mostly about the problem rather than the machine. That's been fine — great, even — but lately I've felt a pull toward understanding what's actually happening underneath. Not just "the garbage collector handles it" but how. Not just "this is slow" but why.

C++ is the answer I keep coming back to. It's everywhere that performance actually matters: trading systems, game engines, embedded hardware, compilers. It forces you to think in terms of memory layout, ownership, and cost. It's not comfortable. That's the point.

This is my learning journal. No polish — just notes, experiments, comparisons, and mini-projects as I go. If you're coming from Python or C# yourself, hopefully some of this is useful.

The Plan

Work through fundamentals comparing against Python and C# — the mental model I already have. Then mini-projects: small enough to finish, complex enough to be interesting. The tabs above show the same concepts implemented across all three languages. That's the format I'll follow for each topic.

Python — the baseline

Seven years in. This is the language I think in. Dynamically typed, garbage collected, expressive. Using it here as the reference point.

Hello World

# As minimal as it gets
print("Hello, world!")

Variables & Types

# Dynamically typed — no declarations needed
name    = "Mish"
age     = 38
balance = 1042.75
active  = True

print(type(balance))  # <class 'float'>

Functions

def compound_interest(principal: float, rate: float, years: int) -> float:
    """Return total value after compound interest."""
    return principal * (1 + rate) ** years

result = compound_interest(1000.0, 0.05, 10)
print(f"£{result:.2f}")  # £1628.89

Classes

class Portfolio:
    def __init__(self, name: str, cash: float):
        self.name = name
        self.cash = cash

    def deposit(self, amount: float) -> None:
        self.cash += amount

    def __repr__(self) -> str:
        return f"{self.name}: £{self.cash:.2f}"

p = Portfolio("ISA", 5000.0)
p.deposit(1000.0)
print(p)  # ISA: £6000.00

Memory

# Python — you never think about memory.
# The garbage collector handles everything.

values = [1.0, 2.0, 3.0]  # heap-allocated, GC-managed
ref    = values             # both point to the same list
ref.append(4.0)

print(values)  # [1.0, 2.0, 3.0, 4.0] — same object
Python's simplicity here is a feature — but it hides a lot. That's exactly what I want to pull back.

C# — eleven years of muscle memory

Statically typed, garbage collected, strong OOP. Closer to C++ in structure but still fully managed. A useful middle ground.

Hello World

// Needs a class and a namespace
using System;

class Program
{
    static void Main()
    {
        Console.WriteLine("Hello, world!");
    }
}

Variables & Types

// Statically typed, or use var for inference
string  name    = "Mish";
int     age     = 38;
double  balance = 1042.75;
bool    active  = true;

// Type inferred at compile time
var rate = 0.04;

Functions

// Return type declared before name
static double CompoundInterest(double principal, double rate, int years)
{
    return principal * Math.Pow(1 + rate, years);
}

double result = CompoundInterest(1000.0, 0.05, 10);
Console.WriteLine($"£{result:F2}");  // £1628.89

Classes

public class Portfolio
{
    public string Name { get; }
    public double Cash { get; private set; }

    public Portfolio(string name, double cash)
    {
        Name = name;
        Cash = cash;
    }

    public void Deposit(double amount) => Cash += amount;

    public override string ToString() => $"{Name}: £{Cash:F2}";
}

var p = new Portfolio("ISA", 5000.0);
p.Deposit(1000.0);
Console.WriteLine(p);  // ISA: £6000.00

Memory

// Reference type: lives on the heap, GC-managed
var values = new List<double> { 1.0, 2.0, 3.0 };
var reference = values;  // same object
reference.Add(4.0);
Console.WriteLine(values.Count);  // 4

// Value type: copied on assignment
int a = 10;
int b = a;
b = 99;
Console.WriteLine(a);  // still 10
C# distinguishes reference vs value types — closer to C++ thinking, but the GC still handles cleanup.

C++ — day one

No runtime. No garbage collector. You decide what lives where, and you're responsible for it. That's the deal.

Hello World

// Explicit includes, explicit return
#include <iostream>

int main() {
    std::cout << "Hello, world!" << std::endl;
    return 0;
}
Nothing is magic — iostream must be included explicitly. std::cout is the standard output stream; << pushes characters onto it.

Variables & Types

#include <string>

std::string name    = "Mish";
int         age     = 38;
double      balance = 1042.75;
bool        active  = true;

// auto infers at compile time — not dynamic like Python
auto rate = 0.04;  // deduced as double
auto looks like Python duck typing but isn't — the type is locked in at compile time. The compiler knows; you just don't have to write it out.

Functions

#include <cmath>
#include <iostream>
#include <iomanip>

double compound_interest(double principal, double rate, int years) {
    return principal * std::pow(1.0 + rate, years);
}

int main() {
    double result = compound_interest(1000.0, 0.05, 10);
    std::cout << std::fixed << std::setprecision(2)
              << "£" << result << std::endl;  // £1628.89
    return 0;
}
Formatting output requires <iomanip>. Python's f-strings are already something I miss.

Classes

#include <string>
#include <iostream>
#include <iomanip>

class Portfolio {
public:
    // Initialiser list — constructs members in place
    Portfolio(std::string name, double cash)
        : name_(name), cash_(cash) {}

    void deposit(double amount) { cash_ += amount; }

    void print() const {
        std::cout << name_ << ": £"
                  << std::fixed << std::setprecision(2)
                  << cash_ << std::endl;
    }

private:
    std::string name_;  // trailing underscore = private member convention
    double      cash_;
};

int main() {
    Portfolio p("ISA", 5000.0);
    p.deposit(1000.0);
    p.print();  // ISA: £6000.00
    return 0;
}
The initialiser list : name_(name), cash_(cash) constructs members in place rather than assigning after construction — matters for performance with complex types.

Memory & Pointers

#include <iostream>
#include <memory>
#include <vector>

int main() {
    // Stack allocation — fast, cleaned up automatically
    int x = 42;

    // Pointer — holds the memory address of x
    int* ptr = &x;
    std::cout << "address: " << ptr  << std::endl;
    std::cout << "value:   " << *ptr << std::endl;  // dereference

    // Reference — an alias, not a copy
    int& ref = x;
    ref = 99;
    std::cout << x << std::endl;  // 99 — same variable

    // Heap allocation — smart pointers, not raw new/delete
    auto values = std::make_unique<std::vector<double>>();
    values->push_back(1.0);
    values->push_back(2.0);
    // unique_ptr cleans itself up when it goes out of scope

    return 0;
}
&x gives the address of x. *ptr dereferences it — follows the address to the value. Modern C++ prefers unique_ptr / shared_ptr over raw new/delete — control without the footguns.
