Python · 7 yrs C# · 11 yrs C++ · day 1

I've been writing code professionally for nearly two decades. Eleven years in C# — enterprise software, financial systems, the whole stack. Then Python pulled me in, and for the last seven years that's been my daily language: data pipelines, quant models, automation, analysis.

Both are managed languages. Both handle memory for you. Both let you think mostly about the problem rather than the machine. That's been fine — great, even — but lately I've felt a pull toward understanding what's actually happening underneath. Not just "the garbage collector handles it" but how. Not just "this is slow" but why.

C++ is the answer I keep coming back to. It's everywhere that performance actually matters: trading systems, game engines, embedded hardware, compilers. It forces you to think in terms of memory layout, ownership, and cost. It's not comfortable. That's the point.

This is my learning journal. No polish — just notes, experiments, comparisons, and mini-projects as I go. If you're coming from Python or C# yourself, hopefully some of this is useful.

The Plan

I'm going to work through fundamentals first, always comparing against Python and C# since that's the mental model I already have. Then mini-projects — things small enough to finish, complex enough to be interesting. Everything documented here as I go.

Below are the first comparisons: the same concepts across all three languages.

Hello World

Hello World
# Python — as minimal as it gets
print("Hello, world!")
// C# — needs a class and a namespace
using System;

class Program
{
    static void Main()
    {
        Console.WriteLine("Hello, world!");
    }
}
// C++ — explicit includes, explicit return
#include <iostream>

int main() {
    std::cout << "Hello, world!" << std::endl;
    return 0;
}
💡 C++ makes you include standard library headers explicitly. Nothing is magic. std::cout is the standard output stream — you're literally pushing characters onto it with <<.

Variables & Types

Variables & Types
# Python — dynamically typed, no declarations
name    = "Mish"
age     = 38
balance = 1042.75
active  = True

print(type(balance))  # <class 'float'>
// C# — statically typed, or use var for inference
string  name    = "Mish";

double  balance = 1042.75;
bool    active  = true;

// or inferred:
var rate = 0.04;
// C++ — statically typed, size matters
#include <string>

std::string name    = "Mish";
int         age     = 38;
double      balance = 1042.75;
bool        active  = true;

// auto keyword infers type at compile time
auto rate = 0.04;  // deduced as double
💡 C++ auto looks like Python's dynamic typing but isn't — the type is resolved at compile time, not runtime. The compiler knows; you just don't have to spell it out.

Functions

Functions
def compound_interest(principal: float, rate: float, years: int) -> float:
    """Return total value after compound interest."""
    return principal * (1 + rate) ** years

result = compound_interest(1000.0, 0.05, 10)
print(f"£{result:.2f}")  # £1628.89
// Return type declared before name
static double CompoundInterest(double principal, double rate, int years)
{
    return principal * Math.Pow(1 + rate, years);
}

double result = CompoundInterest(1000.0, 0.05, 10);
Console.WriteLine($"£{result:F2}");  // £1628.89
#include <cmath>
#include <iostream>
#include <iomanip>

// Return type, then name, then typed parameters
double compound_interest(double principal, double rate, int years) {
    return principal * std::pow(1.0 + rate, years);
}

int main() {
    double result = compound_interest(1000.0, 0.05, 10);
    std::cout << std::fixed << std::setprecision(2)
              << "£" << result << std::endl;
    return 0;
}
💡 C++ feels like C# here — explicit types everywhere — but formatting output requires <iomanip>. Python's f-strings are spoiling.

Classes / OOP

Classes
class Portfolio:
    def __init__(self, name: str, cash: float):
        self.name = name
        self.cash = cash

    def deposit(self, amount: float) -> None:
        self.cash += amount

    def __repr__(self) -> str:
        return f"{self.name}: £{self.cash:.2f}"

p = Portfolio("ISA", 5000.0)
p.deposit(1000.0)
print(p)  # ISA: £6000.00
public class Portfolio
{
    public string Name    { get; }
    public double Cash    { get; private set; }

    public Portfolio(string name, double cash)
    {
        Name = name;
        Cash = cash;
    }

    public void Deposit(double amount) => Cash += amount;

    public override string ToString() =>
        $"{Name}: £{Cash:F2}";
}

var p = new Portfolio("ISA", 5000.0);
p.Deposit(1000.0);
Console.WriteLine(p);  // ISA: £6000.00
#include <string>
#include <iostream>
#include <iomanip>

class Portfolio {
public:
    // Constructor
    Portfolio(std::string name, double cash)
        : name_(name), cash_(cash) {}

    void deposit(double amount) { cash_ += amount; }

    void print() const {
        std::cout << name_ << ": £"
                  << std::fixed << std::setprecision(2)
                  << cash_ << std::endl;
    }

private:
    std::string name_;
    double      cash_;
};

int main() {
    Portfolio p("ISA", 5000.0);
    p.deposit(1000.0);
    p.print();  // ISA: £6000.00
    return 0;
}
💡 The initialiser list : name_(name), cash_(cash) is idiomatic C++ — it constructs members directly rather than assigning after construction. Also note the trailing underscore on private members — a common convention.

Memory & Pointers

This is the big one — the thing neither Python nor C# exposes. In both of those languages, memory management is handled for you. In C++, you can manage it yourself. You often should.

Memory
# Python — you never think about memory
# The garbage collector handles everything

values = [1.0, 2.0, 3.0]  # heap-allocated, GC-managed
ref    = values             # both point to the same list
ref.append(4.0)

print(values)  # [1.0, 2.0, 3.0, 4.0] — same object
// C# — reference vs value types, but still GC-managed
// Reference type: lives on the heap
var values = new List<double> { 1.0, 2.0, 3.0 };
var reference = values;  // same object
reference.Add(4.0);
Console.WriteLine(values.Count);  // 4

// Value type: copied
int a = 10;
int b = a;  // copy
b = 99;
Console.WriteLine(a);  // still 10
#include <iostream>
#include <vector>

int main() {
    // Stack allocation — fast, auto-cleaned up
    int x = 42;

    // Pointer — holds the memory address of x
    int* ptr = &x;
    std::cout << "address: " << ptr  << std::endl;
    std::cout << "value:   " << *ptr << std::endl;  // dereference

    // Reference — alias, not a copy
    int& ref = x;
    ref = 99;
    std::cout << x << std::endl;  // 99 — same variable

    // Heap allocation — manual, use smart pointers in modern C++
    auto values = std::make_unique<std::vector<double>>();
    values->push_back(1.0);
    values->push_back(2.0);
    // unique_ptr cleans itself up — no delete needed

    return 0;
}
💡 &x gives the address of x. *ptr dereferences the pointer — follows the address to get the value. References (int&) are safer aliases. Modern C++ uses unique_ptr / shared_ptr instead of raw new/delete — so you get control without the footguns.
