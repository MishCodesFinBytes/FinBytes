Two maps, very different performance

C++ gives you two associative containers: std::map (sorted, tree-based, O(log n)) and std::unordered_map (hash-based, O(1) average). Python's dict and C#'s Dictionary<K,V> are both hash maps — but C++ makes you choose explicitly.

How a hash map works

A hash function maps a key to a bucket index. The value is stored in that bucket. On lookup, hash the key, go to that bucket, check for a match. O(1) on average — assuming a good hash and low load factor.

Collisions

Two keys can hash to the same bucket. C++ std::unordered_map uses chaining — each bucket holds a linked list of entries. Open addressing (used by many other hash maps, including Python's dict) stores colliding entries elsewhere in the table.

Load factor

Ratio of elements to buckets. As it grows, collisions increase and performance degrades. std::unordered_map rehashes (doubles bucket count) when load factor exceeds a threshold (default 1.0). Like vector reallocation — invalidates iterators.

When to use which

  • std::unordered_map — fast lookup by key, order doesn't matter
  • std::map — need sorted iteration, or range queries by key
  • For finance: instrument lookup, position tracking → unordered_map. Order book price levels → map (need sorted iteration from best bid/ask).

Python dict — open addressing hash map

Python's dict uses open addressing with pseudo-random probing. Since Python 3.7 it also preserves insertion order. It's highly optimised — generally faster than std::unordered_map for string keys due to interned strings.

from typing import Any
import sys

# Basic usage
positions: dict[str, float] = {}
positions["AAPL"] = 100.0
positions["MSFT"] = 200.0
positions["GOOGL"] = 50.0

# O(1) average lookup
if "AAPL" in positions:
    print(positions["AAPL"])  # 100.0

# Memory size
print(sys.getsizeof(positions))  # ~200 bytes for 3 items

defaultdict and Counter

from collections import defaultdict, Counter

# defaultdict — auto-initialise missing keys
trades: defaultdict[str, list] = defaultdict(list)
trades["AAPL"].append({"price": 150.0, "qty": 100})
trades["AAPL"].append({"price": 151.0, "qty": 200})

# Counter — frequency map
symbols = ["AAPL", "MSFT", "AAPL", "GOOGL", "AAPL"]
counts = Counter(symbols)
print(counts.most_common(2))  # [('AAPL', 3), ('MSFT', 1)]
Python dict's load factor is ~0.66 — it rehashes when 2/3 full. C++ std::unordered_map default is 1.0. Python's tighter threshold gives better performance at the cost of more memory.

C# Dictionary<K,V> — open addressing

C#'s Dictionary<K,V> uses open addressing with a power-of-two bucket count. Strong generic support — no boxing for value type keys unlike Java's HashMap.

var positions = new Dictionary<string, double>();
positions["AAPL"]  = 100.0;
positions["MSFT"]  = 200.0;

// TryGetValue — safe lookup without exception
if (positions.TryGetValue("AAPL", out double qty))
    Console.WriteLine(qty);  // 100.0

// Pre-sizing
var large = new Dictionary<string, double>(capacity: 10_000);

// Concurrent access
using System.Collections.Concurrent;
var concurrent = new ConcurrentDictionary<string, double>();

SortedDictionary — like std::map

// Sorted by key — O(log n) operations
var orderBook = new SortedDictionary<double, int>();
orderBook[100.0] = 1000;  // price -> qty
orderBook[99.5]  = 500;
orderBook[100.5] = 750;

// Iterate in price order
foreach (var (price, qty) in orderBook)
    Console.WriteLine($"{price}: {qty}");

C++ — unordered_map and map

Basic unordered_map usage

#include <unordered_map>
#include <string>
#include <iostream>

int main() {
    std::unordered_map<std::string, double> positions;

    // Insert
    positions["AAPL"]  = 100.0;
    positions["MSFT"]  = 200.0;
    positions.emplace("GOOGL", 50.0);  // constructed in-place

    // Lookup — operator[] inserts default if missing!
    std::cout << positions["AAPL"] << "\n";  // 100.0

    // Safe lookup
    auto it = positions.find("TSLA");
    if (it != positions.end())
        std::cout << it->second << "\n";
    else
        std::cout << "not found\n";

    // at() throws if missing
    try {
        std::cout << positions.at("TSLA") << "\n";
    } catch (const std::out_of_range&) {
        std::cout << "not found\n";
    }
    return 0;
}
💡 Avoid operator[] for lookups — it inserts a default value if the key doesn't exist, which is usually a bug. Prefer find() or at().

Pre-reserving buckets

#include <unordered_map>

// Reserve buckets upfront to avoid rehashing
std::unordered_map<std::string, double> positions;
positions.reserve(10'000);     // allocate for 10k entries
positions.max_load_factor(0.5); // rehash at 50% — faster lookups, more memory

// Check current state
std::cout << positions.bucket_count() << "\n";
std::cout << positions.load_factor()  << "\n";

std::map — sorted, O(log n)

#include <map>
#include <iostream>

int main() {
    // Order book: price -> quantity (sorted by price)
    std::map<double, int> bids;
    bids[99.5]  = 500;
    bids[100.0] = 1000;
    bids[98.0]  = 250;

    // Iterates in ascending price order
    for (auto& [price, qty] : bids)
        std::cout << price << " -> " << qty << "\n";

    // Best bid = last element (highest price)
    auto best_bid = bids.rbegin();
    std::cout << "Best bid: " << best_bid->first << "\n";
    return 0;
}

Custom hash for numeric keys

#include <unordered_map>

// Integer keys are fast — default hash is identity
std::unordered_map<int, double> instrument_prices;
instrument_prices[12345] = 100.5;  // instrument ID -> price

// For composite keys, provide a custom hash
struct InstrumentKey {
    int venue_id;
    int symbol_id;
    bool operator==(const InstrumentKey& o) const {
        return venue_id == o.venue_id && symbol_id == o.symbol_id;
    }
};

struct InstrumentHash {
    std::size_t operator()(const InstrumentKey& k) const {
        return std::hash<int>{}(k.venue_id) ^
               (std::hash<int>{}(k.symbol_id) << 16);
    }
};

std::unordered_map<InstrumentKey, double, InstrumentHash> prices;
prices[{1, 42}] = 100.5;
