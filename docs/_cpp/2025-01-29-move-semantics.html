The problem move semantics solves

Imagine you have a large price series — a vector of a million doubles. You want to return it from a function. Without move semantics, C++ would copy all million doubles. With move semantics, it steals the underlying buffer — an O(1) operation regardless of size.

lvalues and rvalues

Every expression in C++ is either an lvalue (has an address, can appear on the left of =) or an rvalue (temporary, about to be destroyed). Move semantics let you detect rvalues and steal their resources instead of copying them.

The move constructor

Alongside the copy constructor, modern C++ classes can define a move constructor that takes an rvalue reference (T&&). Instead of duplicating data, it transfers ownership of the internal buffer from the source to the destination, then nulls out the source.

std::move

std::move doesn't move anything — it casts an lvalue to an rvalue reference, telling the compiler "I'm done with this, you can steal from it". The actual move happens in the move constructor or move assignment operator.

Why this matters for quant code

Returning large containers from functions, inserting into vectors of vectors, passing time series between components — all of these are O(1) with move semantics instead of O(n). Essential for any code that manipulates large financial data structures.

Python — everything is a reference

Python doesn't have value semantics for objects — assignment always copies a reference, never the data. There's no concept of move because there's no copying of object contents to begin with.

import sys

prices = [1.0, 2.0, 3.0]

# "assignment" copies the reference, not the data
prices2 = prices
prices2.append(4.0)
print(prices)   # [1.0, 2.0, 3.0, 4.0] — same object!

# To actually copy, use copy() or slicing
import copy
prices3 = prices.copy()       # shallow copy
prices4 = copy.deepcopy(prices)  # deep copy

Returning large data from functions

def load_prices(path: str) -> list[float]:
    prices = []
    with open(path) as f:
        for line in f:
            prices.append(float(line.strip()))
    return prices  # returns reference — no copy

result = load_prices("data.csv")  # O(1) — just a reference
Python's reference semantics mean returning large objects is always O(1) — you're just passing a pointer. C++ value semantics mean you get a copy by default, which is why move semantics matter.

C# — reference types are always "moved"

Reference types in C# work like Python — assignment copies a reference. Value types (structs) are copied by value, but C# has no explicit move semantics. The GC handles cleanup.

// Reference type — assignment copies reference
var prices = new List<double> { 1.0, 2.0, 3.0 };
var prices2 = prices;  // same object
prices2.Add(4.0);
Console.WriteLine(prices.Count);  // 4 — same list

// Value type — copied by value
struct Point { public double X, Y; }
var p1 = new Point { X = 1.0, Y = 2.0 };
var p2 = p1;  // copy
p2.X = 99.0;
Console.WriteLine(p1.X);  // 1.0 — independent copy
C# 7+ added ref return and ref local for avoiding copies in performance-critical code — the same motivation as C++ move semantics, but more limited.

C++ — move semantics in detail

The cost of copying

#include <vector>
#include <chrono>
#include <iostream>
#include <numeric>

std::vector<double> make_prices(int n) {
    std::vector<double> v(n);
    std::iota(v.begin(), v.end(), 0.0);
    return v;  // NRVO: compiler likely elides the copy entirely
}

int main() {
    // Move: buffer ownership transferred, O(1)
    auto prices = make_prices(1'000'000);

    // Explicit copy: duplicates all 1M doubles, O(n)
    auto copy = prices;

    // Explicit move: transfers ownership, O(1), prices is now empty
    auto moved = std::move(prices);
    std::cout << prices.size()  << "\n";  // 0 — emptied
    std::cout << moved.size()   << "\n";  // 1000000
    return 0;
}

Writing a moveable class

#include <cstring>
#include <utility>
#include <iostream>

class PriceSeries {
public:
    explicit PriceSeries(int n)
        : data_(new double[n]), size_(n)
    {
        std::cout << "construct(" << n << ")\n";
    }

    // Copy constructor — duplicates data
    PriceSeries(const PriceSeries& other)
        : data_(new double[other.size_]), size_(other.size_)
    {
        std::memcpy(data_, other.data_, size_ * sizeof(double));
        std::cout << "copy(" << size_ << ")\n";
    }

    // Move constructor — steals the buffer
    PriceSeries(PriceSeries&& other) noexcept
        : data_(other.data_), size_(other.size_)
    {
        other.data_ = nullptr;  // null out the source
        other.size_ = 0;
        std::cout << "move\n";
    }

    ~PriceSeries() { delete[] data_; }

    int    size()           const { return size_; }
    double operator[](int i) const { return data_[i]; }

private:
    double* data_;
    int     size_;
};

int main() {
    PriceSeries s1(1000);         // construct(1000)
    PriceSeries s2 = s1;          // copy(1000)
    PriceSeries s3 = std::move(s1); // move
    std::cout << s1.size() << "\n"; // 0
    std::cout << s3.size() << "\n"; // 1000
    return 0;
}

Move semantics with STL containers

#include <vector>
#include <string>
#include <iostream>

int main() {
    std::vector<std::vector<double>> matrix;

    for (int i = 0; i < 100; ++i) {
        std::vector<double> row(1000, i * 1.0);

        // push_back(row)        — copies row (expensive)
        // push_back(std::move(row)) — moves row (free)
        matrix.push_back(std::move(row));  // O(1) per row
    }

    std::cout << matrix.size() << "\n";  // 100
    return 0;
}
💡 Mark move constructors noexcept. Standard containers like std::vector will only use your move constructor during reallocation if it's marked noexcept — otherwise they fall back to copying for exception safety.

Perfect forwarding — passing moves through

#include <utility>
#include <vector>

template<typename T>
void insert_into(std::vector<T>& v, T&& item) {
    v.push_back(std::forward<T>(item));  // preserves lvalue/rvalue-ness
}

int main() {
    std::vector<std::vector<double>> store;
    std::vector<double> prices = {1.0, 2.0, 3.0};

    insert_into(store, std::move(prices));  // moves into vector
    return 0;
}
