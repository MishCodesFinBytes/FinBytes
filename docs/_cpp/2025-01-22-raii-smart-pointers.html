The problem with manual memory

Raw new and delete are error-prone. Forget to delete and you leak. Delete twice and you crash. Delete while someone else holds a pointer and you corrupt memory. C++ solved this — elegantly — with RAII.

RAII — Resource Acquisition Is Initialisation

The core idea: tie a resource's lifetime to an object's lifetime. When the object is constructed, acquire the resource. When the object is destroyed (goes out of scope), release it. The destructor is the guarantee.

This is the C++ equivalent of Python's context managers (with) or C#'s IDisposable / using — but automatic, not manual.

Smart pointers

The standard library ships three smart pointer types that implement RAII for heap memory:

  • unique_ptr — sole ownership. One owner, auto-deleted when it goes out of scope.
  • shared_ptr — shared ownership. Reference-counted, deleted when the last owner goes away.
  • weak_ptr — non-owning observer of a shared_ptr. Breaks circular references.

The rule of thumb

Prefer unique_ptr by default. Use shared_ptr only when you genuinely need shared ownership. Never use raw new/delete in modern C++ — use make_unique and make_shared instead.

Python — reference counting + GC

Python uses reference counting as its primary memory management. When an object's reference count drops to zero, it's immediately deallocated. A cyclic GC handles circular references that reference counting can't catch.

import sys

x = [1, 2, 3]
print(sys.getrefcount(x))  # 2 (x + the getrefcount arg)

y = x                       # ref count -> 3
del x                       # ref count -> 2
del y                       # ref count -> 0 -> deallocated

Context managers — Python's RAII equivalent

# with statement guarantees __exit__ is called
with open("data.csv") as f:
    data = f.read()
# file closed here — even if an exception was raised

# Custom context manager
from contextlib import contextmanager

@contextmanager
def timed_block(label: str):
    import time
    t0 = time.perf_counter()
    try:
        yield
    finally:
        elapsed = time.perf_counter() - t0
        print(f"{label}: {elapsed*1000:.2f}ms")

with timed_block("pricing"):
    result = price_option(S=100, K=100, T=1, r=0.05, sigma=0.2)
Python's with is explicit — you have to remember to use it. C++ RAII is implicit — destruction happens automatically when the object leaves scope, no matter how (normal return, exception, early return).

C# — IDisposable and using

C# uses a GC but provides IDisposable for deterministic resource cleanup. The using statement calls Dispose() at the end of the block — equivalent to Python's with.

// IDisposable pattern
public class PriceStream : IDisposable
{
    private readonly StreamReader _reader;

    public PriceStream(string path)
    {
        _reader = new StreamReader(path);
    }

    public string? ReadNext() => _reader.ReadLine();

    public void Dispose()
    {
        _reader.Dispose();
        GC.SuppressFinalize(this);
    }
}

// using guarantees Dispose() is called
using var stream = new PriceStream("prices.csv");
var line = stream.ReadNext();
C#'s using is syntactic sugar for a try/finally block. C++ destructors are guaranteed to run — no try/finally needed, no forgetting to add using.

C++ — RAII and smart pointers

RAII with a custom class

#include <fstream>
#include <stdexcept>
#include <string>

class PriceStream {
public:
    explicit PriceStream(const std::string& path)
        : file_(path)
    {
        if (!file_.is_open())
            throw std::runtime_error("Cannot open: " + path);
    }

    ~PriceStream() {
        // destructor — called automatically when object goes out of scope
        // ifstream closes itself, but showing the pattern
        file_.close();
    }

    std::string read_next() {
        std::string line;
        std::getline(file_, line);
        return line;
    }

private:
    std::ifstream file_;
};

void process() {
    PriceStream ps("prices.csv");
    auto line = ps.read_next();
    // ps destroyed here — file closed automatically
    // even if an exception is thrown above
}

unique_ptr — sole ownership

#include <memory>
#include <iostream>

struct OrderBook {
    explicit OrderBook(int id) : id_(id) {
        std::cout << "OrderBook " << id_ << " created\n";
    }
    ~OrderBook() {
        std::cout << "OrderBook " << id_ << " destroyed\n";
    }
    int id_;
};

void demo_unique() {
    auto book = std::make_unique<OrderBook>(42);
    // auto book2 = book;  // ERROR — can't copy a unique_ptr
    auto book2 = std::move(book);  // OK — ownership transferred
    // book is now null; book2 owns the object
}  // book2 destroyed here -> "OrderBook 42 destroyed"

shared_ptr — shared ownership

#include <memory>
#include <iostream>

void demo_shared() {
    auto book = std::make_shared<OrderBook>(1);
    std::cout << "count: " << book.use_count() << "\n";  // 1

    {
        auto book2 = book;  // shared — ref count -> 2
        std::cout << "count: " << book.use_count() << "\n";  // 2
    }  // book2 destroyed — ref count -> 1

    std::cout << "count: " << book.use_count() << "\n";  // 1
}  // book destroyed — ref count -> 0 -> "OrderBook 1 destroyed"

weak_ptr — breaking cycles

#include <memory>

struct Node {
    std::shared_ptr<Node> next;
    std::weak_ptr<Node>   prev;  // weak to avoid cycle
    int value;
};

void demo_weak() {
    auto a = std::make_shared<Node>();
    auto b = std::make_shared<Node>();
    a->next = b;
    b->prev = a;  // weak_ptr — doesn't increment ref count
    // If prev were shared_ptr, a and b would never be freed
}
💡 Use make_unique and make_shared — never bare new. make_shared allocates the object and control block in one allocation, which is faster and more cache-friendly than shared_ptr<T>(new T).
