Why atomics exist

A mutex protects a critical section by blocking other threads. For simple operations like incrementing a counter or reading a flag, this is overkill. Atomics give you thread-safe operations on single values without any blocking — implemented using hardware instructions (LOCK prefix on x86, LDXR/STXR on ARM).

The C++ memory model

Before C++11, the standard said nothing about threads — you were relying on platform behaviour. C++11 introduced a formal memory model: rules about when writes by one thread become visible to other threads. Without these rules, the compiler and CPU are free to reorder instructions in ways that break multithreaded code.

Memory ordering

  • memory_order_relaxed — no ordering guarantees. Just atomicity. Use for statistics counters.
  • memory_order_acquire / memory_order_release — the most useful pair. Release stores synchronise with acquire loads — everything written before a release is visible after an acquire on the same variable.
  • memory_order_seq_cst — sequential consistency. Strongest guarantee, default. Slowest on weakly-ordered architectures (ARM).

When to use atomics vs mutexes

Atomics for single values: flags, counters, sequence numbers. Mutexes for compound state: updating multiple related variables that must stay consistent together.

Python — GIL provides implicit atomicity

The GIL means most simple Python operations are already effectively atomic at the bytecode level. But this is an implementation detail of CPython, not a guarantee — and it doesn't help with multiprocessing.

import threading

# This looks like a data race but works in CPython due to the GIL
counter = 0
lock = threading.Lock()

def increment_safe(n: int) -> None:
    global counter
    for _ in range(n):
        with lock:  # still needed for correctness across implementations
            counter += 1

threads = [threading.Thread(target=increment_safe, args=(10_000,)) for _ in range(8)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)  # 80000
Python has no equivalent of C++ atomics because the GIL provides a coarser guarantee. In C++, without atomics or mutexes, concurrent access to a plain int is undefined behaviour — the compiler can assume it won't happen and optimise accordingly.

C# — Interlocked class

C# provides atomic operations via the Interlocked class and the volatile keyword. Interlocked maps directly to hardware atomic instructions, same as C++ atomics.

using System.Threading;

int counter = 0;

// Atomic increment — no mutex needed
Interlocked.Increment(ref counter);
Interlocked.Add(ref counter, 5);

// Compare-and-swap
int oldVal = Interlocked.CompareExchange(ref counter, newValue: 10, comparand: 5);
// Sets counter to 10 only if it was 5; returns old value

// Atomic flag
int flag = 0;
bool wasSet = Interlocked.CompareExchange(ref flag, 1, 0) == 0;

// volatile — visibility guarantee, no atomicity
volatile bool done = false;
// One thread sets done = true; another reads it
// volatile ensures the write is visible, but doesn't prevent data races on compound ops

C++ atomics

Atomic counter

#include <atomic>
#include <thread>
#include <vector>
#include <iostream>

std::atomic<int> counter{0};

void increment(int n) {
    for (int i = 0; i < n; ++i)
        counter.fetch_add(1, std::memory_order_relaxed);
        // relaxed: no ordering constraint — just atomicity
        // fine for a counter where we only read the total at the end
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 8; ++i)
        threads.emplace_back(increment, 10'000);
    for (auto& t : threads) t.join();
    std::cout << counter.load() << "\n";  // always 80000
    return 0;
}

Acquire/release — the common pattern

#include <atomic>
#include <thread>
#include <string>
#include <iostream>

std::string data;
std::atomic<bool> ready{false};

void producer() {
    data = "Hello from producer";     // write data
    ready.store(true, std::memory_order_release);  // signal: data is ready
    // Everything written before the release store is visible
    // to any thread that acquires the same atomic
}

void consumer() {
    while (!ready.load(std::memory_order_acquire))
        ;  // spin until ready
    // At this point, data is guaranteed to be "Hello from producer"
    // The acquire load synchronises with the release store
    std::cout << data << "\n";
}

int main() {
    std::thread p(producer);
    std::thread c(consumer);
    p.join(); c.join();
    return 0;
}

Compare-and-swap — the basis of lock-free code

#include <atomic>
#include <iostream>

// Lock-free max update
std::atomic<double> max_price{0.0};

void update_max(double price) {
    double current = max_price.load(std::memory_order_relaxed);
    while (price > current) {
        // Try to swap: only succeeds if max_price still equals current
        if (max_price.compare_exchange_weak(
                current, price,
                std::memory_order_release,
                std::memory_order_relaxed))
            break;  // succeeded
        // Failed: current updated with latest value, retry
    }
}

int main() {
    std::vector<std::thread> threads;
    for (double p : {101.5, 99.0, 103.2, 98.5, 104.0})
        threads.emplace_back(update_max, p);
    for (auto& t : threads) t.join();
    std::cout << max_price.load() << "\n";  // 104.0
    return 0;
}
💡 compare_exchange_weak can spuriously fail on some architectures (ARM) — that's fine in a loop. Use compare_exchange_strong only when you can't retry (e.g. in an if statement without a loop).

Sequence number pattern — common in quant systems

#include <atomic>
#include <cstdint>

// Assign monotonically increasing IDs without a mutex
class OrderIdGenerator {
public:
    uint64_t next() {
        return seq_.fetch_add(1, std::memory_order_relaxed);
    }
private:
    std::atomic<uint64_t> seq_{1};
};

// One global instance — all threads get unique IDs
OrderIdGenerator order_ids;
auto id = order_ids.next();
