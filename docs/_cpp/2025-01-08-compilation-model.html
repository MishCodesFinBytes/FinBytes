Why this matters

In Python you run a file. In C# you compile a project. In C++ you compile translation units and link them together — and understanding that pipeline explains half the errors you'll see as a beginner.

This post covers what actually happens when you build a C++ program: preprocessing, compilation, assembly, linking — and why the header/.cpp split exists.

The pipeline

  • Preprocessor — expands #include, #define, #ifdef. Pure text substitution.
  • Compiler — turns each .cpp (translation unit) into an object file .o
  • Linker — combines object files into an executable, resolves symbol references

The header / .cpp split

Headers (.h) declare — they tell the compiler what exists. Source files (.cpp) define — they provide the actual implementation. This split lets multiple translation units share declarations without duplicating code.

One Definition Rule (ODR)

Every symbol must be defined exactly once across all translation units. Define a function in a header (without inline) and include it in two .cpp files — the linker will complain about duplicate symbols. This is one of the most common C++ beginner errors.

Build systems

You don't invoke the compiler manually for real projects. CMake is the standard — it generates build files for your platform. A minimal CMakeLists.txt is shown in the C++ tab.

Python — no compilation step

Python is interpreted. There's no separate compile step — the interpreter reads and executes source directly. .pyc files are a bytecode cache, not compiled machine code.

# Just run it — no headers, no linking
# module_a.py
def greet(name: str) -> str:
    return f"Hello, {name}"

# main.py
from module_a import greet  # runtime import

print(greet("Mish"))  # Hello, Mish
Python imports happen at runtime. If module_a.py doesn't exist, you find out when the line executes — not before.

Circular imports

# Python handles circular imports (awkwardly)
# a.py imports b.py which imports a.py
# Usually fixed by restructuring or lazy imports
from typing import TYPE_CHECKING
if TYPE_CHECKING:
    from b import SomeType  # only for type checker, not runtime

C# — project-based compilation

C# compiles an entire project at once into an assembly (.dll or .exe). No header files — the compiler reads all .cs files in the project and resolves references automatically.

// Greeter.cs — declaration and definition in one file
public class Greeter
{
    public string Greet(string name) => $"Hello, {name}";
}

// Program.cs — just reference the class, no include needed
var g = new Greeter();
Console.WriteLine(g.Greet("Mish"));  // Hello, Mish
C# has no equivalent of the header/.cpp split. The compiler resolves all symbols within the project automatically. Cross-project references use assembly references in the .csproj file.

Project file (csproj)

<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net8.0</TargetFramework>
  </PropertyGroup>
  <ItemGroup>
    <PackageReference Include="SomeLib" Version="1.0.0" />
  </ItemGroup>
</Project>

C++ — headers, translation units, linking

Header file — declaration only

// greeter.h — declares the interface
#pragma once  // include guard: prevents double-inclusion

#include <string>

class Greeter {
public:
    std::string greet(const std::string& name) const;
};

Source file — definition

// greeter.cpp — defines the implementation
#include "greeter.h"

std::string Greeter::greet(const std::string& name) const {
    return "Hello, " + name;
}

Main

// main.cpp
#include <iostream>
#include "greeter.h"  // only the declaration needed here

int main() {
    Greeter g;
    std::cout << g.greet("Mish") << std::endl;
    return 0;
}

Building manually

# Compile each translation unit to an object file
g++ -c greeter.cpp -o greeter.o
g++ -c main.cpp    -o main.o

# Link into an executable
g++ greeter.o main.o -o app

./app  # Hello, Mish

CMakeLists.txt — the standard build system

cmake_minimum_required(VERSION 3.20)
project(MyApp)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

add_executable(app
    main.cpp
    greeter.cpp
)
# Build with CMake
mkdir build && cd build
cmake ..
cmake --build .
💡 #pragma once is a non-standard but universally supported include guard. The traditional alternative is #ifndef GREETER_H / #define GREETER_H / ... / #endif — same effect, more typing.

inline — the ODR escape hatch

// In a header — safe because of inline
inline int square(int x) { return x * x; }

// Without inline, including this in two .cpp files
// would cause a linker error: duplicate symbol 'square'
💡 inline doesn't mean the function gets inlined by the compiler — it means "multiple definitions are allowed, they're all the same". The compiler decides whether to actually inline based on its own heuristics.
