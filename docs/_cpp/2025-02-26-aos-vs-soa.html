The cache is the bottleneck

Modern CPUs are so fast that they're almost always waiting for memory. A cache miss costs ~100ns — hundreds of cycles stalled. For tight loops over large datasets (pricing, risk, backtesting), memory layout determines performance more than algorithmic complexity.

How CPU caches work

When you access a memory address, the CPU fetches not just that address but an entire cache line — typically 64 bytes. If your next access is within that 64-byte window, it's essentially free. If it's elsewhere in RAM, you take another 100ns penalty.

Array of Structs (AoS)

Store an array of structs: [{price, volume, time}, {price, volume, time}, ...]. Natural and readable. When you loop over prices only, you load the volume and time fields too — wasted cache bandwidth.

Struct of Arrays (SoA)

Store each field in its own array: prices[], volumes[], times[]. When you loop over prices, you only load prices — full cache line utilisation. Common in high-performance quant, graphics, and game engines.

When to use which

  • AoS — when you always access all fields together (e.g. processing a single tick with all its fields)
  • SoA — when you loop over one or two fields at a time (e.g. computing VWAP over a million ticks)

Python — objects are always AoS (badly)

Python objects are heap-allocated and scattered in memory. A list of tick objects is a list of pointers, each pointing to an object elsewhere on the heap. Cache performance is poor regardless of access pattern.

from dataclasses import dataclass

# AoS — natural Python style
@dataclass
class Tick:
    price:  float
    volume: int
    time:   float

ticks = [Tick(100.0 + i*0.01, 1000, i*0.001) for i in range(1_000_000)]

# Looping over prices — loads entire Tick object for each price
import time
t0 = time.perf_counter()
total = sum(t.price for t in ticks)
print(f"AoS: {(time.perf_counter()-t0)*1000:.1f}ms")

NumPy — SoA under the hood

import numpy as np

# SoA — each field is a contiguous array
N = 1_000_000
prices  = np.random.uniform(99, 101, N)
volumes = np.random.randint(100, 10000, N)
times   = np.linspace(0, 3600, N)

# Vectorised loop — all prices in one contiguous block
t0 = time.perf_counter()
total = prices.sum()
print(f"SoA (numpy): {(time.perf_counter()-t0)*1000:.1f}ms")
# Typically 5-20x faster than the object loop above
NumPy's performance advantage over Python lists isn't just "C vs Python" — it's also the memory layout. Contiguous arrays of raw doubles are cache-friendly. Lists of Python objects are not.

C# — structs give AoS with value semantics

C# structs stored in arrays give contiguous memory layout — similar to C++ AoS. For SoA, you maintain separate arrays manually, same as C++.

// AoS with struct — contiguous in memory
struct Tick {
    public double Price;
    public int    Volume;
    public double Time;
}

var ticks = new Tick[1_000_000];
for (int i = 0; i < ticks.Length; i++)
    ticks[i] = new Tick { Price = 100.0 + i * 0.01, Volume = 1000, Time = i * 0.001 };

// Loop over prices — loads entire Tick per iteration
double sum = 0;
for (int i = 0; i < ticks.Length; i++)
    sum += ticks[i].Price;

SoA in C#

// SoA — separate arrays per field
var prices  = new double[1_000_000];
var volumes = new int[1_000_000];
var times   = new double[1_000_000];

// Loop over prices only — full cache efficiency
double sum = 0;
for (int i = 0; i < prices.Length; i++)
    sum += prices[i];

C++ — AoS vs SoA with benchmarks

AoS layout

#include <vector>
#include <chrono>
#include <numeric>
#include <iostream>

struct Tick {
    double price;   // 8 bytes
    int    volume;  // 4 bytes
    double time;    // 8 bytes
    int    flags;   // 4 bytes
};  // 24 bytes per tick — price is 1 of 3 doubles per cache line

using AoS = std::vector<Tick>;

double sum_prices_aos(const AoS& ticks) {
    double sum = 0;
    for (const auto& t : ticks)
        sum += t.price;  // loads 24 bytes, uses 8
    return sum;
}

SoA layout

struct TickStore {  // Struct of Arrays
    std::vector<double> prices;   // all prices contiguous
    std::vector<int>    volumes;
    std::vector<double> times;
    std::vector<int>    flags;

    explicit TickStore(std::size_t n) {
        prices.resize(n);
        volumes.resize(n);
        times.resize(n);
        flags.resize(n);
    }

    std::size_t size() const { return prices.size(); }
};

double sum_prices_soa(const TickStore& store) {
    double sum = 0;
    for (double p : store.prices)  // 100% cache utilisation
        sum += p;
    return sum;

    // Or with SIMD-friendly std::reduce:
    // return std::reduce(store.prices.begin(), store.prices.end(), 0.0);
}

Benchmark

int main() {
    constexpr int N = 1'000'000;

    // Populate AoS
    AoS ticks(N);
    for (int i = 0; i < N; ++i)
        ticks[i] = {100.0 + i*0.01, 1000, i*0.001, 0};

    // Populate SoA
    TickStore store(N);
    for (int i = 0; i < N; ++i) {
        store.prices[i]  = 100.0 + i*0.01;
        store.volumes[i] = 1000;
        store.times[i]   = i*0.001;
    }

    auto bench = [](auto label, auto fn, auto& data) {
        auto t0 = std::chrono::high_resolution_clock::now();
        volatile double s = 0;
        for (int iter = 0; iter < 100; ++iter) s += fn(data);
        auto t1 = std::chrono::high_resolution_clock::now();
        auto us = std::chrono::duration_cast<std::chrono::microseconds>(t1-t0).count();
        std::cout << label << ": " << us/100 << "us avg\n";
    };

    bench("AoS", sum_prices_aos, ticks);
    bench("SoA", sum_prices_soa, store);
    // Typical result: SoA is 2-4x faster
    return 0;
}
💡 With SIMD auto-vectorisation enabled (-O2 or -O3), the SoA advantage grows further. The compiler can vectorise a loop over contiguous doubles using AVX2 — processing 4 doubles per instruction. AoS interleaving prevents this.

Hybrid — hot/cold splitting

// Split frequently-accessed fields from rarely-accessed ones
struct TickHot {   // accessed every tick
    double price;
    int    volume;
};

struct TickCold {  // accessed only on signals
    double time;
    int    flags;
    char   source[8];
};

std::vector<TickHot>  hot_data(N);
std::vector<TickCold> cold_data(N);
// hot_data fits more ticks per cache line — better for the main loop
