Three approaches

C++ gives you three ways to handle errors, and the right choice depends on context:

  • Exceptions — the traditional C++ approach. Good for truly exceptional conditions. Banned in many low-latency paths because they can add latency in the unhappy path.
  • Error codes — return an int or enum indicating success/failure. Fast, explicit, easy to ignore (which is the problem).
  • std::expected<T, E> (C++23) — return either a value or an error. Forces the caller to handle both cases. No overhead on the success path. The modern approach.

In quant/finance code

Exceptions for setup/teardown and file I/O. std::expected for pricing and risk functions that can fail with meaningful errors (bad input, convergence failure). Error codes for C-style APIs and hot paths where you need maximum predictability.

Python — exceptions are idiomatic

from typing import Union
from dataclasses import dataclass
from enum import Enum

# Standard Python: exceptions
def price_option(S: float, K: float, T: float) -> float:
    if S <= 0: raise ValueError(f"S must be positive, got {S}")
    if T <= 0: raise ValueError(f"T must be positive, got {T}")
    return max(S - K, 0)  # simplified

# Result type — Python doesn't enforce this but it's a useful pattern
class PricingError(Enum):
    INVALID_INPUT   = "invalid_input"
    NO_CONVERGENCE  = "no_convergence"

@dataclass
class Ok:
    value: float

@dataclass
class Err:
    error: PricingError
    message: str

Result = Union[Ok, Err]

def price_option_safe(S: float, K: float, T: float) -> Result:
    if S <= 0:
        return Err(PricingError.INVALID_INPUT, f"S must be positive, got {S}")
    return Ok(max(S - K, 0))

match price_option_safe(100, 95, 1):
    case Ok(value=v):   print(f"Price: {v}")
    case Err(message=m): print(f"Error: {m}")

C# — exceptions and Result patterns

// Standard: exceptions
double PriceOption(double S, double K, double T) {
    if (S <= 0) throw new ArgumentException($"S must be positive, got {S}");
    if (T <= 0) throw new ArgumentException($"T must be positive, got {T}");
    return Math.Max(S - K, 0);
}

// Result pattern (manual, or use FluentResults / ErrorOr libraries)
public record Result<T>(T? Value, string? Error) {
    public bool IsOk  => Error is null;
    public bool IsErr => Error is not null;

    public static Result<T> Ok(T value)       => new(value, null);
    public static Result<T> Err(string error)  => new(default, error);
}

Result<double> PriceOptionSafe(double S, double K, double T) {
    if (S <= 0) return Result<double>.Err($"S must be positive, got {S}");
    return Result<double>.Ok(Math.Max(S - K, 0));
}

var result = PriceOptionSafe(100, 95, 1);
if (result.IsOk) Console.WriteLine($"Price: {result.Value}");
else             Console.WriteLine($"Error: {result.Error}");

C++ error handling approaches

Exceptions — traditional

#include <stdexcept>
#include <cmath>

double price_option(double S, double K, double T) {
    if (S <= 0) throw std::invalid_argument("S must be positive");
    if (T <= 0) throw std::invalid_argument("T must be positive");
    return std::max(S - K, 0.0);
}

try {
    auto px = price_option(-1, 100, 1);
} catch (const std::invalid_argument& e) {
    std::cerr << "Error: " << e.what() << "\n";
}

std::expected — C++23 (recommended for new code)

#include <expected>
#include <string>
#include <cmath>
#include <iostream>

enum class PricingError {
    InvalidInput,
    NoConvergence,
};

std::string to_string(PricingError e) {
    switch (e) {
        case PricingError::InvalidInput:   return "invalid input";
        case PricingError::NoConvergence:  return "no convergence";
    }
    return "unknown";
}

std::expected<double, PricingError>
price_option(double S, double K, double T) {
    if (S <= 0 || T <= 0)
        return std::unexpected(PricingError::InvalidInput);
    return std::max(S - K, 0.0);
}

// Chaining with and_then / transform
auto result = price_option(100, 95, 1)
    .transform([](double px) { return px * 1.0; })  // apply discount
    .and_then([](double px) -> std::expected<double, PricingError> {
        if (px < 0) return std::unexpected(PricingError::InvalidInput);
        return px;
    });

if (result)
    std::cout << "Price: " << *result << "\n";
else
    std::cout << "Error: " << to_string(result.error()) << "\n";
💡 std::expected has zero overhead on the success path — it's just a union of T and E with a flag. Unlike exceptions, there's no stack unwinding cost. The caller is forced to check the result (or explicitly ignore it), unlike error codes which are easy to forget.

Error codes — for C APIs and maximum predictability

// Common in low-latency paths that cannot afford exception machinery
enum class Status : int {
    Ok            = 0,
    InvalidInput  = 1,
    NoConvergence = 2,
};

Status price_option_fast(
    double S, double K, double T,
    double& out_price)  // output parameter
{
    if (S <= 0 || T <= 0) return Status::InvalidInput;
    out_price = std::max(S - K, 0.0);
    return Status::Ok;
}

double px;
if (auto s = price_option_fast(100, 95, 1, px); s == Status::Ok)
    std::cout << px << "\n";
