The thing Python and C# hide from you

Every variable you've ever declared lives somewhere in memory. In Python and C#, the runtime decides where. In C++, you decide — and that decision affects performance dramatically.

There are two regions that matter: the stack and the heap.

The stack

Fast, automatic, limited. When you call a function, a stack frame is pushed — all local variables live there. When the function returns, the frame is popped and that memory is instantly reclaimed. No allocator, no GC, no overhead.

  • Allocation: essentially free (just move the stack pointer)
  • Size: typically 1–8 MB per thread
  • Lifetime: tied to the enclosing scope
  • Cache: extremely cache-friendly — consecutive frames are contiguous

The heap

Flexible, manual, slower. new asks the OS for memory at runtime. The allocator has to find a free block, track it, and eventually reclaim it. That costs time — and heap objects can end up scattered across memory, killing cache performance.

  • Allocation: slow (allocator bookkeeping)
  • Size: limited only by available RAM
  • Lifetime: until you (or a smart pointer) release it
  • Cache: potentially poor — objects may be far apart in memory

Why this matters for quant code

A tight pricing loop that allocates on the heap on every iteration will be orders of magnitude slower than one that works entirely on the stack or pre-allocated arrays. Understanding this is the foundation of performance-aware C++.

Python — everything is on the heap

In CPython, every object — including integers and booleans — is heap-allocated. The garbage collector (reference counting + cyclic GC) manages their lifetimes. You have no control over this.

import sys

# Even a small int is a heap object
x = 42
print(sys.getsizeof(x))   # 28 bytes — not 4!
print(sys.getsizeof(3.14)) # 24 bytes

# A list is a heap object containing pointers to more heap objects
prices = [100.5, 101.0, 99.8]
print(sys.getsizeof(prices))  # 120 bytes for the list shell alone
                               # each float is another 24 bytes on the heap
Python's per-object overhead (reference count, type pointer, GC info) is ~16–28 bytes on top of the actual data. A Python float costs 6x more memory than a C++ double.

NumPy — the escape hatch

import numpy as np

# NumPy arrays store raw doubles in a contiguous C block
prices = np.array([100.5, 101.0, 99.8], dtype=np.float64)
print(prices.itemsize)  # 8 bytes — same as C++ double
print(prices.nbytes)    # 24 bytes total — no per-element overhead

# This is why NumPy is fast: cache-friendly contiguous memory
# Vectorised operations bypass Python's object overhead entirely

C# — value types vs reference types

C# is closer to C++ here. Value types (struct, int, double, bool) live on the stack when declared as local variables. Reference types (class) always live on the heap.

// Value type — lives on the stack
int x = 42;
double price = 100.5;

// Struct — stack allocated when local
struct Tick {
    public double Price;
    public long   Volume;
}
Tick t = new Tick { Price = 100.5, Volume = 1000 };  // stack, no GC

// Reference type — always heap allocated
class Order {
    public double Price { get; set; }
    public int    Qty   { get; set; }
}
var o = new Order { Price = 100.5, Qty = 100 };  // heap, GC managed
C#'s Span<T> and stackalloc allow stack allocation of arrays — a performance feature added specifically for low-latency scenarios like financial systems.

stackalloc for hot paths

// Stack-allocate a buffer — no GC pressure
Span<double> buffer = stackalloc double[256];
for (int i = 0; i < buffer.Length; i++)
    buffer[i] = i * 0.01;

// buffer is freed automatically when the method returns

C++ — you choose where everything lives

Stack allocation

#include <array>
#include <iostream>

void stack_demo() {
    // All of these live on the stack
    int    x      = 42;
    double price  = 100.5;
    char   buf[64] = {};  // 64-byte buffer, zero-initialised

    // Fixed-size array — stack allocated, cache-friendly
    std::array<double, 256> prices;
    prices.fill(0.0);

    // All freed automatically when function returns
}  // <-- stack frame popped here, no allocator involved

Heap allocation

#include <memory>
#include <vector>

void heap_demo() {
    // Raw new/delete — avoid in modern C++
    double* raw = new double[1000];
    delete[] raw;  // must remember this

    // Smart pointer — heap but automatically managed
    auto prices = std::make_unique<double[]>(1000);
    // freed automatically when prices goes out of scope

    // std::vector — heap, but RAII managed
    std::vector<double> v(1000, 0.0);
    // freed automatically
}

Measuring the difference

#include <chrono>
#include <array>
#include <vector>
#include <numeric>
#include <iostream>

constexpr int N = 1'000'000;

double sum_stack() {
    std::array<double, 256> buf;
    std::iota(buf.begin(), buf.end(), 0.0);
    return std::accumulate(buf.begin(), buf.end(), 0.0);
}

double sum_heap() {
    std::vector<double> buf(256);
    std::iota(buf.begin(), buf.end(), 0.0);
    return std::accumulate(buf.begin(), buf.end(), 0.0);
}

int main() {
    auto t0 = std::chrono::high_resolution_clock::now();
    volatile double s = 0;
    for (int i = 0; i < N; ++i) s += sum_stack();
    auto t1 = std::chrono::high_resolution_clock::now();
    for (int i = 0; i < N; ++i) s += sum_heap();
    auto t2 = std::chrono::high_resolution_clock::now();

    auto ms = [](auto d) {
        return std::chrono::duration_cast<std::chrono::milliseconds>(d).count();
    };
    std::cout << "Stack: " << ms(t1-t0) << "ms\n";
    std::cout << "Heap:  " << ms(t2-t1) << "ms\n";
    return 0;
}
💡 On a typical run the heap version is 3–10x slower for small buffers — not because memory access is slower, but because the allocator has to find and track free blocks on every iteration. Pre-allocate once and reuse wherever possible.

The golden rule for quant code

// Bad — allocates on every tick
void price_tick_bad(double price) {
    std::vector<double> buf;       // heap alloc
    buf.push_back(price);
    // ... process
}  // heap dealloc

// Good — allocate once, reuse
class TickProcessor {
    std::vector<double> buf_;      // allocated once in constructor
public:
    TickProcessor() { buf_.reserve(4096); }

    void on_tick(double price) {
        buf_.push_back(price);     // no alloc until capacity exceeded
        // ... process
    }
};
