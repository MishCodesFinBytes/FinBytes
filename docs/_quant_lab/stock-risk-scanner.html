What is the Stock Risk Scanner?

The Stock Risk Scanner is a production-style portfolio risk analysis service. Given a list of stock tickers and portfolio weights, it fetches historical market data, calculates a suite of risk metrics, generates an AI-powered narrative explanation via the Claude API, and persists every result to a PostgreSQL database — all surfaced through a clean FastAPI REST interface.

Request Flow

  1. Request arrives — POST /api/scan validates the payload with Pydantic.
  2. Fetch prices — market_data.py calls yfinance.download for the requested period.
  3. Calculate risk — risk.py computes VaR, CVaR, max drawdown, volatility, and Sharpe Ratio using NumPy.
  4. Generate narrative — narrative.py calls the Claude API (with a formatted fallback if no API key is set).
  5. Persist result — scanner.py writes a ScanRecord to PostgreSQL via SQLAlchemy 2.0 async.
  6. Return result — the caller polls GET /api/scans/{id} until status is complete.

Risk Metrics — The Mathematics

Every metric the scanner produces is grounded in standard quantitative finance. Here is what each one means, how it is calculated, and why it matters.

Portfolio Returns

The daily profit-or-loss of the blended portfolio.

R_p = sum(w_i * ln(P_i,t / P_i,t-1))

For each stock, we compute the log return — the natural log of today's price divided by yesterday's price. We then take a weighted sum using the portfolio weights. Log returns are preferred because they are additive across time and better behaved statistically than simple percentage returns.

Value at Risk (VaR, 95%)

How bad can a typical bad day get?

VaR_95 = percentile(R_p, 5)

VaR at the 95% confidence level is the 5th percentile of the portfolio's daily return distribution. In plain English: on 95% of days, your portfolio will not lose more than this amount. If VaR is -1.8%, you should expect a loss exceeding 1.8% only about one trading day in twenty.

Conditional VaR (CVaR / Expected Shortfall)

When things go bad, how bad on average?

CVaR = mean(R_p where R_p < VaR_95)

CVaR takes the average of all returns that fall below the VaR threshold. It answers the question: "Given that we are in the worst 5% of days, what is the expected loss?" CVaR is a more complete risk measure than VaR because it captures the shape of the tail, not just its cutoff point.

Maximum Drawdown

The worst peak-to-trough decline over the period.

Drawdown_t = (cumulative_t - running_max_t) / running_max_t
Max Drawdown = min(Drawdown_t)

We track the portfolio's cumulative value over time and record the largest percentage drop from any peak to any subsequent trough. This is the worst loss you would have experienced if you bought at a local high and held through the low. A drawdown of -15% means the portfolio fell 15% from its peak before recovering.

Volatility (Annualised)

How much does the portfolio bounce around?

Volatility = std(R_p) * sqrt(252)

Volatility is the annualised standard deviation of daily portfolio returns. We multiply by sqrt(252) because there are approximately 252 trading days per year and variance scales linearly with time (so standard deviation scales with the square root). Higher volatility means more uncertainty — the portfolio can swing further in either direction on any given day.

Sharpe Ratio

Return per unit of risk.

Sharpe = annualised_return / volatility

The Sharpe Ratio normalises return by risk. A portfolio that returns 10% with 5% volatility has a Sharpe of 2.0 — better than one returning 15% with 12% volatility (Sharpe 1.25). Above 1 is considered good; above 2 is excellent. The scanner uses a simplified version without subtracting the risk-free rate, which is appropriate for relative comparisons between scans.

Core Module Walkthrough

The scanner's business logic lives in five Python modules under projects/stock-risk-scanner/src/scanner/.

models.py — Data Contracts

All inputs and outputs are defined as Pydantic v2 models, giving automatic validation and serialisation.

  • ScanRequest — tickers (list of uppercase strings), weights (must sum to 1.0), period (default "1y"). A custom validator ensures len(tickers) == len(weights).
  • RiskMetrics — five float fields: var_pct, cvar_pct, max_drawdown_pct, volatility_pct, sharpe_ratio.
  • ScanResult — combines tickers, weights, a RiskMetrics instance, an AI-generated narrative string, and a timestamp.

risk.py — Metric Calculation

calculate_risk_metrics(prices, weights) -> RiskMetrics

  1. Compute log returns: np.log(prices / prices.shift(1)).dropna()
  2. Weight the returns: returns @ np.array(weights)
  3. VaR: np.percentile(portfolio_returns, 5)
  4. CVaR: mean of returns below VaR
  5. Max drawdown: np.exp(np.cumsum(...)) then track running max
  6. Volatility: std * sqrt(252)
  7. Sharpe: mean * 252 / volatility

market_data.py — Price Fetching

fetch_prices(tickers, period) -> pd.DataFrame

Wraps yfinance.download(). Handles MultiIndex columns for multiple tickers, forward-fills missing values, and raises ValueError for empty data.

narrative.py — AI Commentary

The RiskNarrator class generates plain-English risk summaries.

  • With API key: calls Claude with a senior risk analyst system prompt and formatted metrics.
  • Without API key: returns a deterministic fallback string summarising the five metrics.

scanner.py — Orchestrator

scan(request) -> ScanResult

Calls fetch_prices, then calculate_risk_metrics, then RiskNarrator.generate, assembles a ScanResult and returns it. The FastAPI background task calls this function and writes the result to the database.

Infrastructure

API Endpoints

MethodPathStatusDescription
GET/health200Liveness check
POST/api/scan202Accepts scan request, fires background task
GET/api/scans/{id}200/404Retrieve a single scan result
GET/api/scans200List recent completed scans

POST returns immediately with HTTP 202 and the scan ID. The scan runs asynchronously via asyncio.ensure_future, so the caller polls GET until status transitions from pending to complete (or failed).

Database — ScanRecord

A SQLAlchemy 2.0 async model with 14 columns tracking tickers, weights, period, status, five risk metrics, narrative, error message, and timestamps.

Status flow: pending → complete (with metrics + narrative) or failed (with error message).

Alembic Migrations

Async-compatible env.py using run_sync inside an async context. A single autogenerated initial migration creates the scan_records table.

Docker

Multi-stage Dockerfile: builder installs dependencies, runtime copies only installed packages + app code. docker-compose defines two services: scanner (port 8000) and postgres:16 (port 5432) with a named volume for data persistence.

CI — GitHub Actions

Triggers on push to working branch and PRs to master. Steps: checkout, Python 3.11, pip install, pytest. All 26 tests must pass.

Logging

structlog configured with JSON output and ISO-8601 timestamps. Every scan request, completion, and failure is logged as structured JSON.

Run It Locally

Prerequisites

  • Python 3.11
  • Docker Desktop (running)
  • git

Step 1 — Clone the repository

git clone https://github.com/MishCodesFinBytes/QuantLab.git
cd QuantLab

Step 2 — Start the services

cd projects/stock-risk-scanner
docker compose up --build -d

This builds the scanner image and starts both the FastAPI app (port 8000) and PostgreSQL (port 5432).

Step 3 — Run the database migration

python -m alembic upgrade head

Run from the projects/stock-risk-scanner directory on the host (not inside the container).

Step 4 — Verify the service

curl http://localhost:8000/health

Expected: {"status": "ok"}

Step 5 — Submit a scan

curl -X POST http://localhost:8000/api/scan \
  -H "Content-Type: application/json" \
  -d '{"tickers": ["AAPL", "MSFT"], "weights": [0.6, 0.4]}'

Returns: {"id": 1, "status": "pending"}

Step 6 — Poll for the result

Wait about 10 seconds, then:

curl http://localhost:8000/api/scans/1

Sample Output

{
  "id": 1,
  "status": "complete",
  "tickers": ["AAPL", "MSFT"],
  "weights": [0.6, 0.4],
  "period": "1y",
  "var_pct": -2.14,
  "cvar_pct": -3.19,
  "max_drawdown_pct": -19.37,
  "volatility_pct": 23.89,
  "sharpe_ratio": 0.61,
  "narrative": "Portfolio AAPL, MSFT: VaR -2.14%, CVaR -3.19%, ...",
  "created_at": "2026-04-02T20:39:48Z",
  "completed_at": "2026-04-02T20:39:49Z"
}

Step 7 — Run the test suite

pip install -e ".[dev]"
pytest -v

All 26 tests should pass.

Step 8 — Stop the services

docker compose down
