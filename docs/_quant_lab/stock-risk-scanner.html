What is the Stock Risk Scanner?

The Stock Risk Scanner is a production-style portfolio risk analysis service. Given a list of stock tickers and portfolio weights, it fetches historical market data, calculates a suite of risk metrics, generates an AI-powered narrative explanation via the Claude API, and persists every result to a PostgreSQL database — all surfaced through a clean FastAPI REST interface.

Request Flow

  1. Request arrives — POST /api/scan validates the payload with Pydantic.
  2. Fetch prices — market_data.py calls yfinance.download for the requested period.
  3. Calculate risk — risk.py computes VaR, CVaR, max drawdown, volatility, and Sharpe Ratio using NumPy.
  4. Generate narrative — narrative.py calls the Claude API (with a formatted fallback if no API key is set).
  5. Persist result — scanner.py writes a ScanRecord to PostgreSQL via SQLAlchemy 2.0 async.
  6. Return result — the caller polls GET /api/scans/{id} until status is complete.

Risk Metrics — The Mathematics

Every metric the scanner produces is grounded in standard quantitative finance. Here is what each one means, how it is calculated, and why it matters.

Portfolio Returns

The daily profit-or-loss of the blended portfolio.

Formula:

R_p = ∑(w_i × ln(P_i,t / P_i,t−1))

For each stock, we compute the log return — the natural log of today's price divided by yesterday's price. We then take a weighted sum using the portfolio weights. Log returns are preferred because they are additive across time and better behaved statistically than simple percentage returns.

Value at Risk (VaR, 95%)

How bad can a typical bad day get?

VaR_95 = percentile(R_p, 5)

VaR at the 95% confidence level is the 5th percentile of the portfolio's daily return distribution. In plain English: on 95% of days, your portfolio will not lose more than this amount. If VaR is −1.8%, you should expect a loss exceeding 1.8% only about one trading day in twenty.

Conditional VaR (CVaR / Expected Shortfall)

When things go bad, how bad on average?

CVaR = mean(R_p where R_p < VaR_95)

CVaR takes the average of all returns that fall below the VaR threshold. It answers the question: "Given that we are in the worst 5% of days, what is the expected loss?" CVaR is a more complete risk measure than VaR because it captures the shape of the tail, not just its cutoff point.

Maximum Drawdown

The worst peak-to-trough decline over the period.

Drawdown_t = (cumulative_return_t − running_max_t) / running_max_t
Max Drawdown = min(Drawdown_t)

We track the portfolio's cumulative value over time and record the largest percentage drop from any peak to any subsequent trough. This is the worst loss you would have experienced if you bought at a local high and held through the low. A drawdown of −15% means the portfolio fell 15% from its peak before recovering.

Volatility (Annualised)

How much does the portfolio bounce around?

Volatility = std(R_p) × √252

Volatility is the annualised standard deviation of daily portfolio returns. We multiply by √252 because there are approximately 252 trading days per year and variance scales linearly with time (so standard deviation scales with the square root). Higher volatility means more uncertainty — the portfolio can swing further in either direction on any given day.

Sharpe Ratio

Return per unit of risk.

Sharpe = annualised_return / volatility

The Sharpe Ratio normalises return by risk. A portfolio that returns 10% with 5% volatility has a Sharpe of 2.0 — better than one returning 15% with 12% volatility (Sharpe 1.25). Above 1 is considered good; above 2 is excellent. Note: the scanner uses a simplified version without subtracting the risk-free rate, which is appropriate for relative comparisons between scans.

Core Module Walkthrough

The scanner's business logic lives in five Python modules under projects/stock-risk-scanner/src/scanner/.

models.py — Data Contracts

All inputs and outputs are defined as Pydantic v2 models, giving automatic validation and serialisation.

  • ScanRequest — tickers (list of uppercase strings), weights (must sum to 1.0, each between 0 and 1), period (default "1y", validated against yfinance-accepted strings). A custom validator ensures len(tickers) == len(weights).
  • RiskMetrics — five float fields: var_95, cvar_95, max_drawdown, volatility, sharpe_ratio. All produced by risk.py.
  • ScanResult — combines tickers, weights, a RiskMetrics instance, an AI-generated narrative string, and an ISO-format timestamp. This is the final API response body.

risk.py — Metric Calculation

calculate_risk_metrics(prices: pd.DataFrame, weights: list[float]) -> RiskMetrics

  1. Compute log returns: np.log(prices / prices.shift(1)).dropna()
  2. Weight the returns: returns @ np.array(weights) (matrix multiply)
  3. VaR: np.percentile(portfolio_returns, 5)
  4. CVaR: portfolio_returns[portfolio_returns <= var].mean()
  5. Max drawdown: cumulative product of (1 + returns), then track running max and minimum ratio
  6. Volatility: portfolio_returns.std() * np.sqrt(252)
  7. Sharpe: portfolio_returns.mean() * 252 / volatility

market_data.py — Price Fetching

fetch_prices(tickers: list[str], period: str) -> pd.DataFrame

Wraps yfinance.download(tickers, period=period, auto_adjust=True). When multiple tickers are requested, yfinance returns a MultiIndex DataFrame; the module slices the Close column and renames columns to plain ticker names. Missing values (trading halts, newly listed stocks) are forward-filled with .ffill() and any remaining NaNs dropped, ensuring a clean price matrix for risk.py.

narrative.py — AI Commentary

The RiskNarrator class has a single public method: generate(metrics: RiskMetrics, tickers: list[str]) -> str.

  • With API key: sends a messages.create call to Claude with a system prompt that sets the persona as a senior risk analyst, and a user message containing the formatted metrics. The response is the natural-language narrative returned in the API.
  • Without API key (fallback): returns a deterministic, template-formatted string that summarises the five metrics in plain English — ensuring the service works end-to-end without an Anthropic subscription.

scanner.py — Orchestrator

scan(request: ScanRequest) -> ScanResult

This is the single entry point for a complete scan. It calls fetch_prices, passes the result to calculate_risk_metrics, passes metrics to RiskNarrator.generate, assembles a ScanResult, and returns it. The FastAPI background task calls this function and then writes the result (or any exception) to the database.

Infrastructure

API Endpoints

MethodPathStatusDescription
GET/health200Liveness check — returns {"status": "ok"}
POST/api/scan202Accepts a scan request, creates a pending DB record, fires background task
GET/api/scans/{id}200 / 404Retrieve a single scan result by integer ID
GET/api/scans200List all scan records (most recent first)

The POST endpoint returns immediately with HTTP 202 Accepted and the scan id. The actual scan runs asynchronously via asyncio.ensure_future, so the caller polls the GET endpoint until status transitions from pending to complete (or failed).

Alembic Migrations

The project uses Alembic with an async-compatible env.py — it imports the async engine and uses run_sync inside an async context so Alembic's synchronous migration runner can drive the async SQLAlchemy engine. A single autogenerated initial migration creates the scan_records table.

Docker

The multi-stage Dockerfile has two stages:

  1. builder — installs all Python dependencies into a virtual environment.
  2. runtime — copies only the venv and application source from the builder, keeping the image lean.

docker-compose.yml defines two services: scanner (the FastAPI app, port 8000 exposed) and postgres (postgres:16 image, port 5432 exposed to the host for migrations). A named Docker volume preserves PostgreSQL data across container restarts.

CI — GitHub Actions

A workflow file triggers on push to the working branch and on pull requests to master. Steps: checkout, set up Python 3.11, install dependencies (pip install -e ".[dev]"), run pytest. All 26 tests must pass for the job to succeed.

Logging

The application uses structlog configured with JSON output and ISO-8601 timestamps. Every request, background task start/finish, and exception is logged as a structured JSON line — making logs easy to ingest into any log aggregation tool (e.g., Datadog, CloudWatch, Grafana Loki).

How to Run the Stock Risk Scanner

Prerequisites

  • Python 3.11
  • Docker Desktop (running)
  • git

Step 1 — Clone the repository

git clone https://github.com/MishCodesFinBytes/QuantLab.git
cd QuantLab

Step 2 — Start the services

cd projects/stock-risk-scanner
docker compose up --build -d

This builds the scanner image and starts both the FastAPI app (port 8000) and PostgreSQL (port 5432).

Step 3 — Run the database migration

python -m alembic upgrade head

Run this from the projects/stock-risk-scanner directory on the host (not inside the container). The DATABASE_URL in .env points to localhost:5432, which Docker exposes.

Step 4 — Verify the service is healthy

curl http://localhost:8000/health

Expected response: {"status": "ok"}

Step 5 — Submit a scan

curl -X POST http://localhost:8000/api/scan \
  -H "Content-Type: application/json" \
  -d '{"tickers": ["AAPL", "MSFT"], "weights": [0.6, 0.4]}'

The API returns HTTP 202 immediately with the scan ID:

{"id": 1, "status": "pending"}

Step 6 — Poll for the result

Wait approximately 10 seconds for the background task to complete, then:

curl http://localhost:8000/api/scans/1
